集群架构

MynaDB 采用分离式架构，包含四种不同的节点类型：Meta 节点用于基于 Raft 的共识和元数据协调， Router 节点用于无状态请求路由，Data 节点用于存储和查询执行， 以及可选的 Admin 节点用于集群管理。表级路由使元数据开销保持最小， 同时支持跨数百个 Data 节点的水平扩展。

前提条件：本页假设您已阅读集群快速入门指南，并已有一个运行中的本地集群。 有关生产部署指导，请参阅集群扩缩容。

Meta 节点（Raft 共识）

Meta 节点构成集群的协调骨干。它们使用 Raft 共识协议维护强一致的、可复制的元数据存储。 所有集群状态 — 节点注册、表所有权、分布式锁和领导者选举 — 都通过 Raft 日志条目在所有 Meta 节点间复制。

  • 技术：OpenRaft 0.9 + sled 嵌入式键值数据库用于持久化存储
  • 部署：始终部署奇数个 Meta 节点（3 或 5 个）以维持 Raft 仲裁
  • 容错：N 个 Meta 节点的集群可容忍 ⌊N/2⌋ 个同时故障（例如 3 节点容忍 1 个故障，5 节点容忍 2 个）
  • 一致性：所有写入通过 Raft 领导者；读取可由任何节点提供（可选通过 ReadIndex 实现线性一致读）

Raft 配置

Raft 参数控制领导者选举时序和日志压缩。这些默认值针对低延迟本地网络进行了调优。 对于跨数据中心部署，应按 Meta 节点之间的往返时间比例增加选举超时。

toml 
[meta.raft]
heartbeat_interval_ms = 150          # Leader sends heartbeats at this interval
election_timeout_min_ms = 300        # Minimum election timeout (randomized)
election_timeout_max_ms = 500        # Maximum election timeout (randomized)
snapshot_threshold = 10000           # Compact Raft log after this many entries
max_payload_entries = 300            # Max entries per AppendEntries RPC
replication_lag_threshold = 5000     # Warn if follower is this far behind

# Derived timing:
#   Heartbeat:         150ms
#   Election timeout:  300-500ms (randomized per node)
#   Failover time:     ~500-650ms (election timeout + 1 heartbeat)

状态机

Raft 状态机由 sled 支持，sled 是一个提供崩溃安全持久化存储的嵌入式键值存储。 所有集群元数据按照定义明确的键前缀组织：

键前缀 值类型 描述
/raft/vote Vote 当前任期和投票候选人（Raft 协议状态）
/raft/log/{index} LogEntry 按日志位置索引的 Raft 日志条目；快照后压缩
/raft/snapshot Snapshot 序列化的状态机快照，用于日志压缩和新节点追赶
/nodes/{node_id} NodeInfo 节点注册：角色、状态、模式、地址、租约过期时间、能力
/tables/{catalog}/{schema}/{table} TableMeta 表所有权：所有者节点、分区数、总字节数、版本计数器
/locks/{key} LockEntry 带 TTL 的分布式锁：持有者节点、获取时间、过期时间
/events/{sequence} ClusterEvent 有序事件流，用于缓存失效和 Router 通知
/leader/campaign/{role} LeaderEntry 后台任务的领导者选举（压缩合并、数据保留） — 与 Raft 领导者分开

MetaCommand 类型

对 Meta 状态机的所有变更都表示为 MetaCommand 变体。这些命令被提议到 Raft 领导者， 复制到多数派 Follower，然后应用到状态机。下表列出了每种命令类型：

命令 发起者 描述
RegisterNodeData / Router注册新节点，包含角色、地址和能力。创建带 TTL 的租约。
DeregisterNodeAdmin / Node优雅地从集群中移除节点。如果节点拥有表，则触发表重新分配。
HeartbeatData / Router续期节点租约。包含当前负载指标（CPU、内存、连接数、表数量）。
UpdateNodeStateAdmin / Node在状态之间转换节点（Active、Draining、Retiring 等）。
RegisterTableRouter / Data在元数据目录中注册新表。将初始所有权分配给请求的 Data 节点。
AddTableOwnerRouter为表添加次要所有者（用于备用复制或读扩展）。
RefreshOwnerTTLData续期表的所有权 TTL，防止正常运行期间所有权过期。
RemoveTableOwnerRouter / Admin从表的所有者列表中移除节点（如节点排空或手动重新分配时）。
ReassignTableMeta（内部）将表所有权转移到另一个 Data 节点。由节点故障或负载均衡触发。
AcquireLockData / Router获取带 TTL 的分布式锁。用于压缩合并协调和写入序列化。
ReleaseLockData / Router释放之前获取的锁。持有者租约过期时自动释放。
CampaignData竞选命名的领导者角色（如压缩合并领导者）。每个角色只有一个节点胜出。
PublishData / Meta向事件流发布集群事件，用于 Router 缓存更新和 Admin 通知。
ExpireLeasesMeta（定时器）定期命令，移除过期的节点租约和所有权条目。在 Raft 领导者上每 5 秒运行一次。

领导者选举

Meta 集群中的 Raft 领导者选举是自动的，无需手动干预。当当前领导者不可用时（崩溃、网络分区或优雅关闭）， 剩余的 Follower 检测到心跳缺失并开始新的选举。

选举时间线 
T=0ms     Leader sends last heartbeat
T=150ms   Heartbeat interval passes, no new heartbeat
T=300ms   Follower A's election timeout fires (randomized: 300-500ms)
T=300ms   Follower A increments term, votes for self, sends RequestVote RPCs
T=310ms   Follower B receives RequestVote, grants vote (has not voted this term)
T=310ms   Follower A receives majority vote, becomes new leader
T=320ms   New leader sends heartbeat to establish authority
T=320ms   New leader begins accepting MetaCommand proposals

Total failover time: ~320ms (best case) to ~650ms (worst case)

事件流

Meta 集群发布有序的集群事件流。Router 节点订阅此流以保持本地 ShardCache 更新，无需轮询。 事件分配单调递增的序列号，并保留可配置的时长（默认：5 分钟）。

事件类型 触发者 消费者动作
NodeRegisteredRegisterNodeRouter 将新 Data 节点添加到连接池
NodeStateChangedUpdateNodeState、租约过期Router 将节点标记为排空中或从路由目标中移除
TableOwnershipChangedRegisterTable、ReassignTable、RemoveTableOwnerRouter 使受影响表的 ShardCache 条目失效
LeaderChangedRaft 选举、CampaignAdmin UI 更新领导者指示器；Data 节点检查是否应运行后台任务
LockAcquiredAcquireLock等待同一锁的 Data 节点可以立即退避
LockReleasedReleaseLock、租约过期等待锁的 Data 节点可以尝试获取

Router 节点（请求路由）

Router 节点是所有数据操作的客户端入口点。它们是完全无状态的 — 本地不存储数据， 任何 Router 都可以处理任何请求。这使得它们可以在负载均衡器后面轻松水平扩展。

  • 协议：HTTP（InfluxDB Line Protocol 写入、SQL 查询）、Apache Arrow Flight（高吞吐量二进制）、MySQL 线协议（BI 工具兼容）
  • 路由：基于本地 ShardCache 的表级路由，将表映射到所有者 Data 节点
  • 故障转移：Data 节点故障时自动重试到次要所有者
  • 连接池：维护到所有 Data 节点的持久 Arrow Flight 连接

表级路由

MynaDB 使用表级路由而非文件级或分区级路由。这是一个经过深思熟虑的设计选择， 在路由开销和粒度之间取得平衡。下表比较了三种方法在 100 个数据库、每数据库 100 个表、 每表 365 个日分区、平均 16 个 Parquet 文件/分区条件下的表现：

路由粒度 路由表大小 元数据开销 重平衡成本
文件级 ~58,400,000 条目 非常高（~2.3 GB 内存） 逐文件迁移
分区级 ~3,650,000 条目 高（~146 MB 内存） 逐分区迁移
表级（已选用） ~100,000 条目 低（~4 MB 内存） 逐表重新分配

查询协调

查询路由比写入路由更复杂，因为单个 SQL 查询可能引用由不同 Data 节点拥有的多个表。 Router 实现了扇出/扇入模式：

查询路由步骤 
1. PARSE       Parse the SQL query and extract all referenced table names
               Example: "SELECT * FROM cpu JOIN memory ON cpu.host = memory.host"
               Tables: ["cpu", "memory"]

2. RESOLVE     For each table, look up the owner data node(s) in the ShardCache
               cpu    -> data-1 (primary), data-3 (secondary)
               memory -> data-2 (primary), data-1 (secondary)

3. DETERMINE   Identify the set of unique data nodes that need to participate
               Target nodes: {data-1, data-2}

4. FAN-OUT     Send the full SQL query to each target node via Arrow Flight DoGet
               (parallel, non-blocking)
               data-1: executes locally, returns partial results for cpu + memory
               data-2: executes locally, returns partial results for memory

5. AGGREGATE   Receive RecordBatch streams from all nodes
               Merge partial results (union, then re-apply ORDER BY / LIMIT)

6. RETURN      Stream the final result back to the client
               (Arrow Flight stream, JSON, or MySQL result set)

Data 节点（存储与执行）

Data 节点是集群的主力。它们通过 Arrow Flight 从 Router 节点接收写入，在内存中缓冲数据， 持久化到共享对象存储，并使用 Apache DataFusion 执行 SQL 查询。 每个 Data 节点独立运行 — Data 节点之间没有直接通信。

  • Primary 模式：接受写入并为拥有的表提供查询服务
  • Standby 模式：对备份的表提供只读查询执行（不写入，不使用 WAL）
  • 后台任务：压缩合并、数据保留和缓存刷新（选举的领导者任务仅在一个节点上运行）

存储布局

在共享对象存储（S3/MinIO/OSS）上，Parquet 文件按层次目录结构组织，支持日期分区和节点级命名空间：

S3 存储布局 
{bucket}/
  {catalog}/                              # Database name (e.g., "metrics")
    {schema}/                             # Schema name (e.g., "service_1")
      {table}/                            # Table/measurement (e.g., "cpu")
        date=2026-02-13/                  # Date partition
          node_id=data-1/                 # Node namespace (prevents conflicts)
            L0_1739462400_0001.parquet    # Level 0 (recent flush)
            L0_1739462460_0002.parquet    # Level 0 (another flush)
          node_id=data-2/                 # Different node's files
            L0_1739462400_0001.parquet
        date=2026-02-12/
          node_id=data-1/
            L1_1739376000_0001.parquet    # Level 1 (compacted from L0)
            L2_1739376000_0001.parquet    # Level 2 (compacted from L1)

# File naming convention:
#   L{level}_{timestamp}_{sequence}.parquet
#
# Level 0: Direct flush from DataCache (many small files)
# Level 1: First compaction pass (fewer, larger files)
# Level 2+: Higher compaction levels (fewest, largest files)

压缩合并

压缩合并将小的 L0 Parquet 文件合并为更大的、查询效率更高的高层文件。 MynaDB 使用受 LSM-tree 启发的分层压缩策略：

层级 文件大小（典型） 来源 触发条件
L01-10 MBDataCache 刷新自动（flush_interval_secs）
L150-100 MBL0 压缩当 L0 文件数超过 min_files（默认：5）
L2200-500 MBL1 压缩当 L1 文件数超过阈值
L3+500 MB - 1 GBL2+ 压缩可配置至 max_level（默认：5）

Admin 节点

Admin 节点提供用于监控和操作集群的 Web 管理控制台。它完全是可选的，对数据路径没有影响。Admin 节点：

  • 连接 Meta 集群进行服务发现（发现所有 Router 和 Data 节点）
  • 在 http://<admin-host>:18080 提供 Web UI 进行集群监控
  • 通过 HTTP（非 Arrow Flight）将查询请求转发到各个 Data 节点
  • 显示实时节点健康、表所有权和事件历史
  • 提供交互式 SQL 查询控制台，支持语法高亮和结果可视化
  • 提供存储浏览器，用于探索 Parquet 文件和分区元数据
Admin API 端点 
GET  /api/v1/nodes              # List all registered nodes with health status
GET  /api/v1/nodes/:id          # Get details for a specific node
GET  /api/v1/tables             # List all tables across the cluster
GET  /api/v1/events             # Recent cluster events (paginated)
POST /api/v1/query/:db          # Execute SQL on a specific data node
GET  /api/v1/storage/catalogs   # Browse storage catalogs
GET  /api/v1/storage/info       # Storage backend information
