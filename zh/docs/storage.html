存储

为您的机器提供应有的记忆。MynaDB 以 Apache Parquet 格式存储时序数据，这是一种针对分析查询 优化的列式存储格式。本页介绍存储架构、分区策略和优化技术。

概述

  写入路径                         存储
      |                               |
      v                               v
  +--------+     刷写      +---------------------------+
  | 缓存   | ------------> | Parquet 文件              |
  | (RAM)  |    (30s)      |                           |
  +--------+               | data/                      |
                           | └── mydb/                  |
                           |     └── default/           |
                           |         └── cpu/           |
                           |             └── date=.../  |
                           |                 ├── L0_... |
                           |                 ├── L1_... |
                           |                 └── L2_... |
                           +---------------------------+

核心特性

  • 列式存储 - 只读取需要的列
  • 压缩 - 最高可达 90% 的存储空间缩减
  • 日期分区 - 高效的时间范围查询
  • LSM 层级 - 优化的读写平衡
  • 谓词下推 - 在存储层过滤

Apache Parquet 格式

Apache Parquet 是一种专为高效数据处理 设计的列式存储格式。它是 MynaDB 高性能存储的基础。

为什么选择 Parquet？

🗃

列式布局

数据按列而非按行存储。只访问少数列的查询读取的数据量更少。

📦

压缩

列中相似的值压缩效果好。支持 Snappy、ZSTD、LZ4 等算法。

🔍

谓词下推

文件尾部的统计信息允许跳过不相关的行组。

🌐

生态系统

兼容 Spark、Pandas、DuckDB、DataFusion 等众多工具。

Parquet 文件结构

  +------------------------------------------+
  |              Parquet 文件                 |
  +------------------------------------------+
  |  行组 1                                   |
  |  +--------------------------------------+|
  |  | 列块: timestamp                      ||
  |  | 列块: host                           ||
  |  | 列块: usage                          ||
  |  +--------------------------------------+|
  |                                          |
  |  行组 2                                   |
  |  +--------------------------------------+|
  |  | 列块: timestamp                      ||
  |  | 列块: host                           ||
  |  | 列块: usage                          ||
  |  +--------------------------------------+|
  |                                          |
  |  尾部 (Schema + 统计信息)                 |
  +------------------------------------------+

目录结构

MynaDB 以层级结构组织文件：

data/
├── {catalog}/                        # 数据库名称
│   ├── {schema}/                     # Schema（服务或 "default"）
│   │   └── {table}/                  # Measurement 名称
│   │       └── date=YYYY-MM-DD/      # 日期分区
│   │           ├── L0_xxx.parquet    # Level 0（最近写入）
│   │           ├── L0_yyy.parquet
│   │           ├── L1_aaa.parquet    # Level 1（已压缩）
│   │           └── L2_bbb.parquet    # Level 2（更高压缩）
│   └── _system/
│       └── series_stats/             # 基数追踪
│           └── {catalog}.parquet
│
└── mydb/                             # 示例数据库
    ├── default/
    │   ├── cpu/
    │   │   ├── date=2024-01-15/
    │   │   │   ├── L0_1705312345_abc.parquet
    │   │   │   └── L1_1705398745_def.parquet
    │   │   └── date=2024-01-16/
    │   │       └── L0_1705398800_ghi.parquet
    │   └── memory/
    │       └── date=2024-01-15/
    └── web/
        └── requests/

日期分区

数据根据每个数据点的时间戳自动按日期分区。 这使得时间范围查询能够通过跳过不相关的分区来提高效率。

工作原理

  1. 提取每个数据点的时间戳
  2. 推导日期：YYYY-MM-DD
  3. 将数据写入对应的分区目录
  4. 带有时间过滤条件的查询会跳过不需要的分区
# Data written with timestamp 2024-01-15T10:30:00Z
cpu,host=server01 usage=75.5 1705315800000000000

# Stored in:
data/mydb/default/cpu/date=2024-01-15/L0_xxx.parquet

# Query only reads date=2024-01-15 partition
SELECT * FROM cpu
WHERE time >= '2024-01-15' AND time < '2024-01-16'

分区裁剪

当您按时间过滤时，只扫描相关的分区：

查询 扫描的分区
WHERE time > '2024-01-15' 仅 date=2024-01-15 及之后的分区
WHERE time BETWEEN '2024-01-10' AND '2024-01-12' 仅 3 个分区
SELECT * FROM cpu（无过滤条件） 所有分区

管理分区

# List partitions
curl http://localhost:7000/partitions/mydb/default/cpu

# Delete a specific partition
curl -X DELETE http://localhost:7000/partitions/mydb/default/cpu/2024-01-15

# Delete old partitions (retention)
curl -X POST http://localhost:7000/retention/mydb/default/cpu \
  -H "Content-Type: application/json" \
  -d '{"older_than_days": 30}'

文件命名约定

Parquet 文件遵循编码元数据的命名约定：

L{level}_{timestamp}_{uuid}.parquet

# Examples:
L0_1705312345_a1b2c3d4.parquet    # Level 0, recent write
L1_1705398745_e5f6g7h8.parquet    # Level 1, compacted
L2_1705485145_i9j0k1l2.parquet    # Level 2, more compacted
组成部分 描述
L{level} 压缩级别（L0-L5）
{timestamp} 文件创建时的 Unix 时间戳
{uuid} 防止冲突的唯一标识符

压缩

Parquet 支持多种压缩编码。MynaDB 默认使用 Snappy， 在速度和压缩比之间取得了良好平衡。

压缩编码

编码 压缩比 速度 适用场景
Snappy 良好 快速 默认，均衡
ZSTD 更好 中等 冷数据存储
LZ4 较低 最快 实时查询
GZIP 最佳 较慢 归档

压缩效果

时序数据由于以下原因具有很好的压缩效果：

  • 重复的 Tags - 相同的 host/region 值
  • 相似的值 - 指标变化是渐进的
  • 列式布局 - 相似数据聚集在一起
  • 字典编码 - 字符串去重
📈 典型压缩效果

时序数据使用 Snappy 通常可实现 5-10 倍压缩， 使用 ZSTD 可实现 10-20 倍压缩。100GB 的原始数据集可能压缩到 10GB 或更少。

存储优化

最佳实践

  1. 使用 SSD 存储 - 查询中随机读取很常见
  2. 启用压缩合并 - 减少文件数量并提升查询性能
  3. 设置保留策略 - 防止无限制增长
  4. 监控磁盘使用 - 根据数据采集速率规划容量

估算存储空间

近似存储计算：

# Formula (approximate)
Storage = (points_per_second) × (bytes_per_point) × (seconds) × (1/compression_ratio)

# Example: 10,000 points/sec, 50 bytes/point, 30 days, 10x compression
Storage = 10,000 × 50 × (30 × 86400) / 10
        = 10,000 × 50 × 2,592,000 / 10
        = 129.6 GB

磁盘空间管理

# Check disk usage
du -sh /data/stats/*

# Check by database
du -sh /data/stats/mydb/*

# Check by table
du -sh /data/stats/mydb/default/cpu/*

# Force compaction to reduce file count
curl -X POST http://localhost:7000/compact/mydb

# Apply retention to free space
curl -X POST http://localhost:7000/retention/mydb/default/cpu \
  -H "Content-Type: application/json" \
  -d '{"older_than_days": 7}'

备份与恢复

由于数据以标准 Parquet 文件存储，备份非常简单：

# Backup entire database
tar -czf mydb_backup.tar.gz /data/stats/mydb/

# Backup specific table
tar -czf cpu_backup.tar.gz /data/stats/mydb/default/cpu/

# Restore
tar -xzf mydb_backup.tar.gz -C /data/stats/
