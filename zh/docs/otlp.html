OpenTelemetry (OTLP) 数据接入

MynaDB 原生支持 OpenTelemetry Protocol (OTLP)，可接入指标、日志和链路追踪数据。 您可以直接从应用程序、OpenTelemetry Collector 或任何兼容 OTLP 的 Agent 发送可观测性数据， 无需额外的适配器或转换层。

💡 符合标准

MynaDB 实现了 OpenTelemetry Protocol (OTLP) v1.0 规范， 支持 gRPC 和 HTTP/protobuf 两种传输方式。任何支持 OTLP 的 OpenTelemetry SDK 或 Collector 均可开箱即用地向 MynaDB 发送数据。

支持的端点

MynaDB 公开以下 OTLP 端点用于接收遥测数据：

协议 端点 方法 说明
gRPC :4317 - 原生 gRPC OTLP 接收端，支持所有信号类型（指标、日志、链路追踪）
HTTP /v1/metrics POST HTTP/protobuf 指标接入端点
HTTP /v1/logs POST HTTP/protobuf 日志接入端点
HTTP /v1/traces POST HTTP/protobuf 链路追踪接入端点

配置

在 config.toml 中启用和配置 OTLP 数据接入：

toml 
[otlp]
enabled = true
database = "otel"                  # Target database for OTLP data
grpc_port = 4317                   # gRPC receiver port
http_enabled = true                # Enable HTTP/protobuf endpoints
http_port = 4318                   # HTTP receiver port (on main HTTP server if 0)

# Signal-specific configuration
[otlp.metrics]
enabled = true
table_name = "otel_metrics"        # Table name for metrics data
resource_attributes_as_tags = true # Map resource attributes to tags
include_scope_info = true          # Include instrumentation scope

[otlp.logs]
enabled = true
table_name = "otel_logs"           # Table name for log records
body_field = "body"                # Field name for log body
severity_as_tag = true             # Map severity level to a tag
resource_attributes_as_tags = true

[otlp.traces]
enabled = true
table_name = "otel_traces"         # Table name for trace spans
span_attributes_as_fields = true   # Map span attributes to fields
resource_attributes_as_tags = true
include_events = true              # Store span events as nested data
💡 默认数据库

所有 OTLP 数据都写入由 otlp.database 指定的数据库（默认值：otel）。 在该数据库中，每种信号类型存储在各自配置的 table_name 所指定的表中。

数据映射

MynaDB 自动将 OTLP 数据结构映射为其时序存储模型。 资源属性成为标签，数据点成为对应表中的字段。

指标映射

OTLP 指标被展平存储到 otel_metrics 表中，其 Schema 如下：

列 类型 来源
time TIMESTAMP 数据点时间戳
metric_name STRING (tag) OTLP 指标名称
metric_type STRING (tag) gauge、sum、histogram、summary
value FLOAT 指标值（gauge/sum）
count INTEGER histogram/summary 计数
sum FLOAT histogram/summary 总和
service_name STRING (tag) 资源属性：service.name
host_name STRING (tag) 资源属性：host.name
{attribute} STRING (tag) 其他数据点属性

日志映射

OTLP 日志记录存储在 otel_logs 表中：

列 类型 来源
time TIMESTAMP 日志记录时间戳
severity STRING (tag) 严重级别（INFO、WARN、ERROR 等）
severity_number INTEGER 数值严重级别（1-24）
body STRING 日志消息正文
trace_id STRING 关联的 trace ID（如有）
span_id STRING 关联的 span ID（如有）
service_name STRING (tag) 资源属性：service.name
{attribute} STRING (tag) 其他日志属性

链路追踪映射

OTLP 链路追踪 Span 存储在 otel_traces 表中：

列 类型 来源
time TIMESTAMP Span 开始时间
trace_id STRING (tag) Trace 标识符
span_id STRING Span 标识符
parent_span_id STRING 父 Span 标识符
span_name STRING (tag) 操作名称
span_kind STRING (tag) INTERNAL、SERVER、CLIENT、PRODUCER、CONSUMER
duration_ns INTEGER Span 持续时间（纳秒）
status_code STRING (tag) OK、ERROR、UNSET
status_message STRING 状态描述（错误时）
service_name STRING (tag) 资源属性：service.name
{attribute} STRING Span 属性作为字段

OpenTelemetry Collector

向 MynaDB 发送 OTLP 数据的推荐方式是通过 OpenTelemetry Collector。 在 Collector 管道中将 MynaDB 配置为 OTLP 导出器：

yaml 
# otel-collector-config.yaml
receivers:
  otlp:
    protocols:
      grpc:
        endpoint: 0.0.0.0:4317
      http:
        endpoint: 0.0.0.0:4318

  # Example: scrape Prometheus metrics and forward as OTLP
  prometheus:
    config:
      scrape_configs:
        - job_name: 'node-exporter'
          scrape_interval: 15s
          static_configs:
            - targets: ['localhost:9100']

processors:
  batch:
    timeout: 5s
    send_batch_size: 1000
  memory_limiter:
    check_interval: 1s
    limit_mib: 512
  resource:
    attributes:
      - key: environment
        value: production
        action: upsert

exporters:
  otlp/mynadb:
    endpoint: mynadb-host:4317
    tls:
      insecure: true            # Set to false in production with TLS
    retry_on_failure:
      enabled: true
      initial_interval: 5s
      max_interval: 30s
    sending_queue:
      enabled: true
      num_consumers: 4
      queue_size: 1000

service:
  pipelines:
    metrics:
      receivers: [otlp, prometheus]
      processors: [memory_limiter, batch, resource]
      exporters: [otlp/mynadb]
    logs:
      receivers: [otlp]
      processors: [memory_limiter, batch, resource]
      exporters: [otlp/mynadb]
    traces:
      receivers: [otlp]
      processors: [memory_limiter, batch, resource]
      exporters: [otlp/mynadb]

运行 Collector

bash 
# Docker
docker run -d --name otel-collector \
  -v ./otel-collector-config.yaml:/etc/otelcol/config.yaml \
  -p 4317:4317 \
  -p 4318:4318 \
  otel/opentelemetry-collector-contrib:latest

# Binary
otelcol-contrib --config otel-collector-config.yaml

SDK 直接集成

您也可以使用任何 OpenTelemetry SDK 直接从应用程序发送 OTLP 数据：

python 
# Python example - sending metrics directly to MynaDB
from opentelemetry import metrics
from opentelemetry.sdk.metrics import MeterProvider
from opentelemetry.sdk.metrics.export import PeriodicExportingMetricReader
from opentelemetry.exporter.otlp.proto.grpc.metric_exporter import OTLPMetricExporter

# Configure OTLP exporter pointing to MynaDB
exporter = OTLPMetricExporter(
    endpoint="mynadb-host:4317",
    insecure=True,
)

reader = PeriodicExportingMetricReader(exporter, export_interval_millis=10000)
provider = MeterProvider(metric_readers=[reader])
metrics.set_meter_provider(provider)

# Create and record metrics
meter = metrics.get_meter("myapp")
counter = meter.create_counter("http_requests", description="Total HTTP requests")
histogram = meter.create_histogram("request_duration_ms", description="Request latency")

counter.add(1, {"method": "GET", "path": "/api/users", "status": "200"})
histogram.record(42.5, {"method": "GET", "path": "/api/users"})

查询 OTLP 数据

OTLP 数据接入后，以常规 MynaDB 表的形式存储，可使用标准 SQL 进行查询。 以下是各信号类型的常见查询模式。

查询指标

sql 
-- Average CPU usage per service over the last hour
SELECT
  service_name,
  metric_name,
  time_bucket('5 minutes', time) AS bucket,
  AVG(value) AS avg_value,
  MAX(value) AS max_value
FROM otel.default.otel_metrics
WHERE metric_name = 'system.cpu.utilization'
  AND time > now() - INTERVAL '1 hour'
GROUP BY service_name, metric_name, bucket
ORDER BY bucket DESC;

-- Top 10 services by request count
SELECT
  service_name,
  SUM(value) AS total_requests
FROM otel.default.otel_metrics
WHERE metric_name = 'http_requests'
  AND time > now() - INTERVAL '24 hours'
GROUP BY service_name
ORDER BY total_requests DESC
LIMIT 10;

查询日志

sql 
-- Recent error logs across all services
SELECT
  time,
  service_name,
  severity,
  body
FROM otel.default.otel_logs
WHERE severity = 'ERROR'
  AND time > now() - INTERVAL '1 hour'
ORDER BY time DESC
LIMIT 50;

-- Error rate by service over time
SELECT
  service_name,
  time_bucket('15 minutes', time) AS bucket,
  COUNT(*) FILTER (WHERE severity = 'ERROR') AS errors,
  COUNT(*) AS total,
  ROUND(100.0 * COUNT(*) FILTER (WHERE severity = 'ERROR') / COUNT(*), 2) AS error_pct
FROM otel.default.otel_logs
WHERE time > now() - INTERVAL '6 hours'
GROUP BY service_name, bucket
ORDER BY bucket DESC;

查询链路追踪

sql 
-- Slowest endpoints in the last hour
SELECT
  service_name,
  span_name,
  COUNT(*) AS span_count,
  AVG(duration_ns / 1000000.0) AS avg_ms,
  MAX(duration_ns / 1000000.0) AS max_ms,
  APPROX_PERCENTILE_CONT(duration_ns / 1000000.0, 0.99) AS p99_ms
FROM otel.default.otel_traces
WHERE span_kind = 'SERVER'
  AND time > now() - INTERVAL '1 hour'
GROUP BY service_name, span_name
ORDER BY p99_ms DESC
LIMIT 20;

-- Trace waterfall - all spans for a specific trace
SELECT
  span_name,
  span_kind,
  service_name,
  duration_ns / 1000000.0 AS duration_ms,
  status_code,
  parent_span_id,
  span_id
FROM otel.default.otel_traces
WHERE trace_id = '4bf92f3577b34da6a3ce929d0e0e4736'
ORDER BY time ASC;

-- Error spans by service
SELECT
  service_name,
  span_name,
  COUNT(*) AS error_count,
  MIN(time) AS first_seen,
  MAX(time) AS last_seen
FROM otel.default.otel_traces
WHERE status_code = 'ERROR'
  AND time > now() - INTERVAL '24 hours'
GROUP BY service_name, span_name
ORDER BY error_count DESC;

跨信号关联

sql 
-- Find error logs associated with slow traces
SELECT
  t.trace_id,
  t.span_name,
  t.duration_ns / 1000000.0 AS duration_ms,
  l.severity,
  l.body AS log_message
FROM otel.default.otel_traces t
JOIN otel.default.otel_logs l
  ON t.trace_id = l.trace_id
WHERE t.duration_ns > 5000000000  -- spans longer than 5 seconds
  AND l.severity IN ('ERROR', 'WARN')
  AND t.time > now() - INTERVAL '1 hour'
ORDER BY t.duration_ns DESC
LIMIT 25;
💡 性能提示

为获得最佳查询性能，查询 OTLP 数据时请始终包含时间范围过滤条件 （time > now() - INTERVAL '...'），以利用基于日期的分区裁剪。 service_name 和 metric_name 等标签列已建立索引，过滤效率很高。
