压缩合并

MynaDB 使用 LSM-tree 风格的多级压缩来优化存储效率和查询性能。近期写入产生的小文件会被逐步合并为更大、更高效的文件。

概述

当数据写入 MynaDB 时，首先进入内存缓存，然后以小型 Parquet 文件（Level 0）的形式刷写到磁盘。随着时间推移，这些小文件不断累积，可能会降低查询速度。压缩合并通过将文件合并在一起来解决这个问题。

为什么压缩合并很重要
  • 减少文件数 - 降低文件系统开销
  • 加快查询 - 减少 I/O，提升文件扫描效率
  • 更好的压缩率 - 更大的文件压缩更高效
  • 减少存储 - 消除重复和过时的数据

LSM-Tree 架构

MynaDB 实现了一种受日志结构合并树（LSM-tree）启发的架构来管理 Parquet 文件：

LSM-Tree 数据流 
Write Request
     │
     ▼
┌─────────────┐
│  In-Memory  │  ← Fast writes
│    Cache    │
└─────────────┘
     │ flush (every 30s)
     ▼
┌─────────────┐
│   Level 0   │  ← Many small files (1-10 MB)
│  (L0_*.parquet)
└─────────────┘
     │ compaction (when ≥4 files)
     ▼
┌─────────────┐
│   Level 1   │  ← Fewer, larger files (10-50 MB)
│  (L1_*.parquet)
└─────────────┘
     │ compaction
     ▼
┌─────────────┐
│   Level 2   │  ← Even fewer files (50-200 MB)
│  (L2_*.parquet)
└─────────────┘
     │
     ▼
    ...       ← Up to Level 5

关键特性

特性 说明
写优化 新数据追加到 L0，不修改已有文件
后台合并 压缩异步运行，不阻塞写入
分级结构 每一级包含逐渐更大、更优化的文件
空间放大 通过层级比率和压缩阈值控制

压缩级别

MynaDB 支持最多 6 个级别（L0-L5）：

级别 文件前缀 典型大小 说明
L0 L0_*.parquet 1-10 MB 从缓存直接刷写，大量小文件
L1 L1_*.parquet 10-50 MB 第一级压缩
L2 L2_*.parquet 50-200 MB 第二级压缩
L3 L3_*.parquet 200-500 MB 第三级压缩
L4 L4_*.parquet 500 MB - 1 GB 第四级压缩
L5 L5_*.parquet 1+ GB 最终级别，完全压缩

文件命名规则

# Format: L{level}_{timestamp}_{uuid}.parquet

L0_1703251234567_a1b2c3d4.parquet   # Level 0, recently flushed
L1_1703251300000_e5f6g7h8.parquet   # Level 1, first compaction
L2_1703252000000_i9j0k1l2.parquet   # Level 2, second compaction

压缩策略

MynaDB 支持两种协同工作的压缩策略：

1. LSM 压缩（默认，持续运行）

在后台持续运行，当达到阈值时合并文件。这是主要的压缩机制。

LSM 压缩流程 
# When L0 has 4+ files:
L0_file1.parquet ─┐
L0_file2.parquet ─┼─► Merge ─► L1_merged.parquet
L0_file3.parquet ─┤
L0_file4.parquet ─┘

# When L1 has 4+ files:
L1_file1.parquet ─┐
L1_file2.parquet ─┼─► Merge ─► L2_merged.parquet
L1_file3.parquet ─┤
L1_file4.parquet ─┘

# And so on up to L5...

2. 全量压缩（计划/手动）

将某一级别的所有文件合并为单个文件。适用于激进优化，但更消耗资源。

全量压缩流程 
# Full compaction at L0:
L0_file1.parquet ─┐
L0_file2.parquet ─┤
L0_file3.parquet ─┼─► Merge ALL ─► L1_single.parquet
L0_file4.parquet ─┤
L0_file5.parquet ─┤
L0_file6.parquet ─┘

配置

压缩在 config.toml 中配置：

config.toml 
[compaction]
# Minimum files at a level before compaction triggers
min_files = 4

# Maximum compaction level (L0-L5)
max_level = 5

# Parallel LSM compaction tasks
max_concurrent_compactions = 8

# Parallel full compaction tasks (lower to avoid OOM)
full_compaction_max_concurrent = 2

# Schedule full compaction at specific hours (0-23)
# Example: Run at 2 AM and 2 PM
full_compaction_hours = [2, 14]

# Enable/disable automatic compaction
enabled = true

配置选项说明

选项 默认值 说明
min_files 4 触发压缩所需的同级别文件数
max_level 5 最高压缩级别（L0-L5）
max_concurrent_compactions 8 并行 LSM 压缩工作线程数
full_compaction_max_concurrent 2 并行全量压缩工作线程数（内存密集型）
full_compaction_hours [] 执行计划全量压缩的时间（0-23 小时）
enabled true 启用/禁用自动压缩
内存注意事项

全量压缩会将所有文件加载到内存中进行合并。在内存有限的系统上，请将 full_compaction_max_concurrent 设置为较低的值（1-2），以避免内存溢出错误。

手动压缩

您可以通过 HTTP API 手动触发压缩：

压缩整个数据库

bash 
# Compact all tables in a database
curl -X POST http://localhost:7000/compact/mydb

# Response
{
  "status": "compaction_started",
  "database": "mydb"
}

压缩指定表

bash 
# Compact a specific table
curl -X POST http://localhost:7000/compact/mydb/cpu

# Response
{
  "status": "compaction_started",
  "database": "mydb",
  "table": "cpu"
}

强制全量压缩

bash 
# Force full compaction (merge all files)
curl -X POST http://localhost:7000/compact/mydb?full=true

# Response
{
  "status": "full_compaction_started",
  "database": "mydb"
}

监控压缩

检查文件数量

通过检查每个级别的文件数量来监控压缩进度：

bash 
# Count files per level
find /data/stats/mydb/default/cpu -name "L*.parquet" | \
  sed 's/.*\(L[0-9]\).*/\1/' | sort | uniq -c

# Output:
#   12 L0   ← Many L0 files, compaction needed
#    3 L1
#    1 L2

检查分区文件

bash 
# List partitions and file counts via API
curl http://localhost:7000/partitions/mydb/default/cpu

# Response
{
  "partitions": [
    {
      "date": "2024-01-15",
      "files": 5,
      "size_bytes": 52428800
    },
    {
      "date": "2024-01-14",
      "files": 2,
      "size_bytes": 104857600
    }
  ]
}

服务器日志

压缩事件会被记录到日志中以便监控：

INFO  Compaction completed: mydb.default.cpu L0 → L1 (merged to 1 file, 45MB)

INFO  Full compaction completed: mydb.default.cpu (12 files → 1 file, 234MB)

最佳实践

⚙

调优 min_files

较低的值（2-3）= 更激进的压缩，更高的 CPU 使用率。
较高的值（6-8）= 更少的压缩频率，更多的文件需要扫描。

🕑

计划全量压缩

使用 full_compaction_hours 在非高峰时段运行全量压缩。 例如：[2, 3, 4] 表示凌晨 2-4 点。

💾

监控磁盘空间

压缩合并过程中会临时使用额外的磁盘空间。请为压缩操作保留 20% 的可用空间。

📈

平衡查询与写入

更多压缩 = 更快查询但 CPU 使用率更高。
更少压缩 = 更快写入但查询更慢。

按工作负载推荐的设置

工作负载 min_files max_concurrent 备注
写密集型 6-8 4 减少压缩频率，优先保障写入
读密集型 2-3 8 激进压缩，优化查询性能
均衡型 4 8 默认设置即可
低内存 4 2 限制并发压缩数

压缩与数据保留

压缩和数据保留协同工作，但服务于不同的目的：

  • 压缩合并 - 合并分区内的文件以提高效率
  • 数据保留 - 删除超过阈值的整个分区

当数据保留策略删除旧分区时，也会移除这些分区中的所有已压缩文件。无需对即将删除的数据进行压缩。

提示

如果您设置了较短的保留期限（例如 7 天），可以考虑使用较不激进的压缩设置，因为数据在压缩带来显著收益之前就会被删除。
