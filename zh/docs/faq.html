常见问题

关于 MynaDB 功能和时序数据管理最佳实践的常见问题。

概述

什么是 MynaDB？

MynaDB 是一个用 Rust 编写的高性能时序数据库。它通过 InfluxDB Line Protocol 接收数据，以 Apache Parquet 格式存储，并通过 Apache DataFusion 提供 SQL 查询。它专为指标、监控和分析工作负载而设计。

MynaDB 与 InfluxDB 兼容吗？

MynaDB 使用 InfluxDB Line Protocol 进行数据接入，使其与许多现有的 InfluxDB 客户端和工具兼容。但查询接口使用 SQL 而非 InfluxQL 或 Flux，为大多数用户提供更大的灵活性和更熟悉的体验。

系统要求是什么？

MynaDB 可在 Linux、macOS 和 Windows 上运行，最低要求如下：

  • CPU：建议 2 核以上
  • 内存：最低 512MB，生产环境建议 2GB 以上
  • 磁盘：建议使用 SSD 以获得最佳性能
  • Rust：从源码构建需要 1.70 以上版本

数据与存储

MynaDB 使用什么存储格式？

MynaDB 使用 Apache Parquet 格式存储数据，这是一种针对分析查询优化的列式存储格式。Parquet 为时序数据提供出色的压缩比和快速的查询性能。

数据是如何分区的？

数据按日期分区（date=YYYY-MM-DD），在每个表内进行。这使得时间范围查询高效，并可通过删除整个日期分区来简单管理数据保留。

ServiceAsSchema 和 ServiceAsTable 有什么区别？

策略 Schema 表 适用场景
ServiceAsSchema Service 标签值 Measurement 名称 每个服务有多个 measurement
ServiceAsTable 固定为 "default" Service 标签值 以服务为中心的查询

数据保留多长时间？

数据保留时间可配置。默认情况下数据保留 7 天。您可以在 config.toml 中调整或通过 API 手动触发数据保留： 

# config.toml
[retention]
enabled = true
default_days = 30    # Keep data for 30 days

查询

MynaDB 支持什么查询语言？

MynaDB 通过 Apache DataFusion 支持标准 SQL。包括 SELECT、WHERE、GROUP BY、ORDER BY、JOIN 和聚合函数。详见 SQL 查询指南。

为什么查询新数据前需要调用 /flush？

MynaDB 为了提高性能会将传入数据缓存在内存中。数据会定期刷新到磁盘（默认：每 30 秒）。如果需要立即查询数据，可调用 /flush 立即将缓冲数据持久化。 

# 查询前强制刷新
curl -X POST http://localhost:7000/flush
curl 'http://localhost:7000/query/mydb' -d 'SELECT * FROM cpu LIMIT 10'

如何跨多表查询？

使用标准 SQL JOIN：

SELECT c.host, c.usage, m.available
FROM cpu c
JOIN memory m ON c.host = m.host AND c.time = m.time
WHERE c.time > now() - INTERVAL '1 hour'

时间戳格式是什么？

时间戳以 Unix 纪元以来的纳秒（19 位整数）存储。写入数据时可以省略时间戳，服务器将使用当前时间。

性能

如何提高查询性能？

  • 使用 LIMIT：探索性查询始终添加 LIMIT
  • 按时间过滤：基于时间的过滤器可利用分区
  • 运行压缩合并：减少文件数量，提高扫描效率
  • 使用 Arrow Flight：大结果集比 HTTP 更快

如何提高写入性能？

  • 批量写入：每次请求发送多个数据点
  • 增大刷新间隔：减少磁盘 I/O 频率
  • 使用 SSD 存储：更快的写入和压缩合并

什么是压缩合并，何时应该运行？

压缩合并将小的 Parquet 文件合并为大文件，提高查询性能并减少存储开销。MynaDB 会自动运行压缩合并，但您也可以手动触发： 

# 压缩数据库中的所有表
curl -X POST http://localhost:7000/compact/mydb

# 压缩特定表
curl -X POST http://localhost:7000/compact/mydb/cpu

API

HTTP API 和 Arrow Flight 有什么区别？

特性 HTTP API Arrow Flight
协议 HTTP/JSON gRPC/Arrow
适用场景 简单查询、写入 大结果集
流式传输 否 是
客户端支持 任何 HTTP 客户端 Arrow Flight 客户端
端口 7000（默认） 7001（默认）

可以将 DataFusion CLI 与 MynaDB 一起使用吗？

可以！MynaDB 的 Arrow Flight 接口与 DataFusion CLI 兼容： 

datafusion-cli --host localhost --port 7001

运维

如何备份数据？

由于数据以 Parquet 文件存储，您可以备份整个 data_dir 目录。为保证一致性，请在备份前停止服务或调用 /flush。

可以运行多个 MynaDB 实例吗？

可以，使用高可用模式。配置主从架构以实现数据复制和自动故障转移。详见高可用指南。

如何监控 MynaDB？

使用内置端点：

# 健康检查
curl http://localhost:7000/health

# 缓存和写入统计
curl http://localhost:7000/stats

# 列出表
curl http://localhost:7000/tables/mydb

迁移

可以从 InfluxDB 迁移吗？

可以。由于 MynaDB 使用 InfluxDB Line Protocol，您可以从 InfluxDB 导出数据并重放到 MynaDB。Telegraf 等工具可以在迁移期间同时写入两个数据库。

可以导入现有的 Parquet 文件吗？

可以，将 Parquet 文件放在相应的目录结构中（data_dir/catalog/schema/table/date=YYYY-MM-DD/）并重启服务即可。请确保 schema 与预期格式匹配。
