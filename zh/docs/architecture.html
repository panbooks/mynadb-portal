架构

正如八哥鸟拥有卓越的聆听和记忆能力，MynaDB 采用分层架构构建， 能够捕获基础设施中的每一个信号。它针对高吞吐量数据采集和快速分析查询进行了优化， 本文档将介绍各组件如何协同工作，为您的机器赋予"声音"。

核心组件

HTTP 服务器 (Axum)

HTTP 服务器处理所有 REST API 请求。基于 Axum 构建，提供以下能力：

  • 高性能异步请求处理
  • 连接池和长连接支持
  • 优雅关闭支持
  • Tower 中间件集成
端点 方法 描述
/write/:db POST 写入数据 (Line Protocol)
/query/:db GET/POST 执行 SQL 查询
/flush POST 将缓存刷写到磁盘
/compact/:db POST 触发压缩合并
/tables/:db GET 列出表

Line Protocol 解析器

高性能解析 InfluxDB Line Protocol：

<measurement>,<tag_key>=<tag_value>,... <field_key>=<field_value>,... [timestamp]

# 示例：
cpu,host=server01,region=us-west usage=75.5,temp=45i 1609459200000000000
  • Measurement - 表名（例如 "cpu"、"memory"）
  • Tags - 用于过滤的索引字符串键值对
  • Fields - 数值或字符串值（实际指标数据）
  • Timestamp - 纳秒级 Unix 时间戳（可选，默认为当前时间）

数据缓存与 Memtable

内存缓冲区，批量处理写入以提高效率，并提供即时查询可见性：

  • 以 Arrow RecordBatch 形式在内存中缓冲传入数据
  • 按数据库、Schema、表和日期分区组织
  • 每 60 秒后台刷写（可配置）
  • 通过 /flush 端点手动刷写
  • Memtable：查询引擎直接从缓存读取最新写入的数据
⚡ 即时查询可见性（新功能！）

写入数据通过 Memtable 可立即被查询 - 无需刷写！查询引擎自动合并内存缓存中的最新写入数据和 Parquet 文件中的已刷写数据。

元数据管理器

基于内存的 Catalog 系统，跟踪所有数据库对象：

  • Catalog：数据库级容器
  • Schema：Catalog 内的命名空间（例如服务分组）
  • 表：带统计信息的 Measurement/指标表
  • 分区：基于日期的分区及文件计数
  • 文件：包含行数和时间戳的单个 Parquet 文件
📈 性能影响

有了元数据管理器，查询规划速度提升 10-100 倍。零文件系统扫描 - 所有元数据保存在内存中，并定期持久化为 JSON。

DataFusion 查询引擎

Apache DataFusion 提供 SQL 查询能力：

  • 完整 SQL 支持（SELECT、JOIN、GROUP BY、窗口函数、CTE）
  • 谓词下推到 Parquet 文件
  • 并行查询执行
  • 内存高效的流式处理
  • 基于 Apache Arrow 的列式处理

Parquet 存储

数据以 Apache Parquet 格式存储：

  • 列式布局 - 高效支持分析查询
  • 压缩 - 最高可达 90% 的存储空间缩减
  • 谓词下推 - 在读取时过滤
  • Schema 演进 - 无需重写即可添加列

压缩合并器（LSM 风格）

多级压缩合并优化存储和查询性能：

  写入路径                      压缩合并
      |                             |
      v                             |
  +-------+                         |
  |  L0   |  (最近写入)             |
  +---+---+                         |
      |                             |
      +-----> 4+ 文件时合并 --------+
      |                             |
      v                             |
  +-------+                         |
  |  L1   |  (第一次压缩)           |
  +---+---+                         |
      |                             |
      +-----> 4+ 文件时合并 --------+
      |                             |
      v                             |
  +-------+                         |
  |  L2   |  (第二次压缩)           |
  +---+---+                         |
      |                             |
      ...                           |
      |                             |
      v                             v
  +-------+
  |  L5   |  (最高级别)
  +-------+
  • L0 - 最近写入，许多小文件
  • L1-L5 - 逐级更大的合并文件
  • 当某级别文件达到 4 个以上时触发压缩
  • 并行压缩以提升性能

存储结构

MynaDB 使用层级 Catalog.Schema.Table 结构：

data/
└── {catalog}/                    # 数据库名称
    ├── {schema}/                 # Service 标签值（或 "default"）
    │   └── {table}/              # Measurement 名称
    │       └── date=YYYY-MM-DD/  # 日期分区
    │           ├── L0_xxx.parquet
    │           ├── L1_xxx.parquet
    │           └── L2_xxx.parquet
    └── _system/
        └── series_stats/         # 基数追踪

Schema 策略

两种将数据映射到 Catalog 结构的策略：

ServiceAsSchema（默认）

  • Schema = Service 标签值
  • Table = Measurement 名称
  • 最适合多租户场景

cpu,service=web → mydb.web.cpu

ServiceAsTable

  • Schema = "default"
  • Table = Service 标签值
  • Measurement 存储为列

cpu,service=web → mydb.default.web

Arrow Flight

Arrow Flight 提供高性能 gRPC 流式传输：

  • 列式数据传输，无序列化开销
  • 兼容 DataFusion CLI 和 Arrow 客户端
  • 大数据集的流式结果
  • 默认运行在端口 7001
bash 
# Connect with DataFusion CLI
datafusion-cli --host localhost --port 7001

# Run SQL queries
> SELECT * FROM mydb.default.cpu LIMIT 10;

关键设计决策

为什么选择 Rust？

  • 内存安全 - 无空指针错误或数据竞争
  • 零成本抽象 - 高级代码，C 级性能
  • Async/await - 高效处理数千连接
  • 强类型系统 - 编译时捕获错误

为什么选择 Parquet？

  • 列式格式 - 只读取需要的列
  • 压缩 - 支持 Snappy、ZSTD、LZ4
  • 谓词下推 - 读取前过滤
  • 生态系统 - 兼容 Spark、Pandas、Arrow

为什么选择 DataFusion？

  • 完整 SQL - 无需学习自定义查询语言
  • 可扩展 - 支持自定义函数和优化
  • Arrow 原生 - 零拷贝数据处理
  • 活跃社区 - Apache Arrow 项目的一部分

性能特征

操作 典型延迟 说明
写入（缓存） < 1ms 数据进入内存后即确认
简单查询 1-10ms 单表，索引过滤
聚合查询 10-100ms 取决于数据量
JOIN 查询 50-500ms 取决于表大小
刷写 100-1000ms 取决于缓存大小
压缩合并 (L0→L1) 1-10s 后台运行
