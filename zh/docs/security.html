认证、RBAC 与安全

面向生产部署的全面安全功能。MynaDB 提供内置认证、基于角色的访问控制、TLS 加密和审计日志， 在每一层保护您的时序数据。

🔐 纵深防御

MynaDB 的安全策略采用分层方法：通过认证验证用户身份，通过 RBAC 授权操作， 通过 TLS 加密传输中的数据，通过审计日志记录所有操作。 每一层都可以独立启用，以匹配您的部署需求。

基本认证

MynaDB 支持对所有 API 端点进行基本用户名/密码认证。 启用认证后，每个请求都必须包含有效的凭据。

配置

在 config.toml 中启用认证：

toml 
[auth]
# Enable authentication (default: false)
enabled = true

# Admin user credentials
username = "admin"
password = "your-secure-password"

使用 curl 进行认证

通过 -u 参数传递凭据：

bash 
# Write data with authentication
curl -u admin:your-secure-password \
  -X POST http://localhost:7000/write/mydb \
  -d 'cpu,host=server01 usage=45.2'

# Query data with authentication
curl -u admin:your-secure-password \
  "http://localhost:7000/query/mydb?sql=SELECT+*+FROM+cpu+LIMIT+10"

# Health check (no authentication required)
curl http://localhost:7000/health
⚠ 重要提示

禁用认证时，所有端点对未经认证的访问开放。 在生产环境中务必启用认证。即使启用了认证，/health 端点仍保持无需认证，以支持负载均衡器的健康检查。

基于角色的访问控制（RBAC）

RBAC 在基本认证的基础上扩展了细粒度权限控制。用户被分配角色， 角色控制他们可以执行哪些操作以及可以访问哪些数据库。

角色

MynaDB 定义了三个内置角色，权限级别递增：

角色 描述 典型用途
admin 对所有操作和数据库拥有完全访问权限 数据库管理员、DevOps 工程师
writer 对分配的数据库拥有读写访问权限 应用服务、数据管道、采集器
reader 对分配的数据库拥有只读访问权限 仪表盘用户、分析师、监控工具

权限矩阵

每个角色授予特定的数据库级权限：

权限 reader writer admin
读取数据（SELECT、/query） ✓ ✓ ✓
写入数据（/write） ✗ ✓ ✓
列出表和 Schema ✓ ✓ ✓
触发压缩合并 ✗ ✗ ✓
管理数据保留策略 ✗ ✗ ✓
删除分区 ✗ ✗ ✓
管理用户和角色 ✗ ✗ ✓
刷新缓存 ✗ ✗ ✓
查看系统统计信息 ✗ ✗ ✓

用户管理 API

通过 REST API 管理用户及其角色。所有用户管理端点需要 admin 角色。

创建用户

bash 
# Create a new user with the writer role
curl -u admin:your-secure-password \
  -X POST http://localhost:7000/auth/users \
  -H "Content-Type: application/json" \
  -d '{
    "username": "app_writer",
    "password": "writer-password-123",
    "role": "writer",
    "databases": ["mydb", "metrics"]
  }'

# Response
{
  "status": "ok",
  "message": "User app_writer created with role writer"
}

列出用户

bash 
# List all users
curl -u admin:your-secure-password \
  http://localhost:7000/auth/users

# Response
{
  "users": [
    {
      "username": "admin",
      "role": "admin",
      "databases": ["*"]
    },
    {
      "username": "app_writer",
      "role": "writer",
      "databases": ["mydb", "metrics"]
    },
    {
      "username": "dashboard",
      "role": "reader",
      "databases": ["mydb"]
    }
  ]
}

更新用户角色

bash 
# Promote a user to admin
curl -u admin:your-secure-password \
  -X PUT http://localhost:7000/auth/users/app_writer/role \
  -H "Content-Type: application/json" \
  -d '{
    "role": "admin",
    "databases": ["*"]
  }'

# Response
{
  "status": "ok",
  "message": "User app_writer role updated to admin"
}

删除用户

bash 
# Delete a user
curl -u admin:your-secure-password \
  -X DELETE http://localhost:7000/auth/users/dashboard

# Response
{
  "status": "ok",
  "message": "User dashboard deleted"
}

TLS/SSL 配置

使用 TLS 加密所有客户端-服务器通信。MynaDB 支持标准 TLS 和双向 TLS（mTLS），适用于需要客户端证书验证的环境。

配置

toml 
[tls]
# Enable TLS encryption (default: false)
enabled = true

# Path to the server certificate (PEM format)
cert_file = "/etc/mynadb/certs/server.crt"

# Path to the server private key (PEM format)
key_file = "/etc/mynadb/certs/server.key"

# Path to the Certificate Authority bundle (PEM format)
# Required for mutual TLS (mTLS) client verification
ca_file = "/etc/mynadb/certs/ca.crt"

# Enable mutual TLS - clients must present a valid certificate
# signed by the CA specified in ca_file (default: false)
mutual_auth = false

使用 TLS 连接

bash 
# Connect using TLS with CA certificate verification
curl --cacert /etc/mynadb/certs/ca.crt \
  -u admin:your-secure-password \
  "https://localhost:7000/query/mydb?sql=SELECT+*+FROM+cpu+LIMIT+5"

# Connect with mutual TLS (mTLS) - client certificate required
curl --cacert /etc/mynadb/certs/ca.crt \
  --cert /etc/mynadb/certs/client.crt \
  --key /etc/mynadb/certs/client.key \
  -u admin:your-secure-password \
  "https://localhost:7000/query/mydb?sql=SELECT+*+FROM+cpu+LIMIT+5"

# Write data over TLS
curl --cacert /etc/mynadb/certs/ca.crt \
  -u admin:your-secure-password \
  -X POST https://localhost:7000/write/mydb \
  -d 'cpu,host=server01 usage=45.2'

TLS 选项

选项 类型 默认值 描述
enabled bool false 启用 TLS 加密
cert_file string - 服务器证书路径（PEM 格式）
key_file string - 服务器私钥路径（PEM 格式）
ca_file string - CA 证书包路径（PEM 格式）
mutual_auth bool false 要求客户端证书（mTLS）
🔑 证书生成

用于开发和测试，可以使用 OpenSSL 生成自签名证书： 

openssl req -x509 -newkey rsa:4096 -keyout server.key -out server.crt -days 365 -nodes -subj "/CN=localhost"

生产环境建议使用受信任的证书颁发机构或内部 PKI（如 HashiCorp Vault、cert-manager 或 AWS ACM）签发的证书。

审计日志

审计日志记录所有经过认证的操作，用于合规、取证和安全监控。 每条记录捕获用户、操作、目标资源、时间戳和结果。

配置

toml 
[audit]
# Enable audit logging (default: false)
enabled = true

# Path to the audit log file
log_file = "/var/log/mynadb/audit.log"

# Log rotation policy: "daily", "size", or "never"
rotation = "daily"

# Maximum number of rotated log files to retain
max_files = 90

审计日志选项

选项 类型 默认值 描述
enabled bool false 启用审计日志
log_file string /var/log/mynadb/audit.log 审计日志文件路径
rotation string daily 轮转策略（daily、size、never）
max_files u32 90 保留的轮转文件数量

📈 审计日志分析

审计日志使用结构化 JSON 格式（每行一条记录），便于导入 Elasticsearch、Loki 或 Splunk 等日志聚合工具。 也可以使用标准工具直接查询审计日志： 

# Find all failed authentication attempts
jq 'select(.action == "AUTH_FAILURE")' /var/log/mynadb/audit.log

# Count operations per user
jq -r '.user' /var/log/mynadb/audit.log | sort | uniq -c | sort -rn

API 认证

启用认证后，客户端必须在每个请求中提供凭据。 MynaDB 支持两种传递认证凭据的方式。

方式一：Basic Auth 请求头

通过标准 HTTP Authorization 请求头发送凭据，使用 Base64 编码的 username:password：

bash 
# Using curl -u (automatically sets Authorization header)
curl -u admin:your-secure-password \
  "http://localhost:7000/query/mydb?sql=SELECT+1"

# Equivalent explicit header
curl -H "Authorization: Basic YWRtaW46eW91ci1zZWN1cmUtcGFzc3dvcmQ=" \
  "http://localhost:7000/query/mydb?sql=SELECT+1"

方式二：Token 查询参数

对于设置请求头不便的环境（如浏览器工具或简单监控脚本）， 可以通过查询参数传递 Token：

bash 
# Pass token as a query parameter
curl "http://localhost:7000/query/mydb?sql=SELECT+1&token=your-api-token"

# Write data with token parameter
curl -X POST "http://localhost:7000/write/mydb?token=your-api-token" \
  -d 'cpu,host=server01 usage=72.5'
⚠ Token 安全

查询参数中的 Token 可能出现在服务器访问日志和浏览器历史记录中。 生产环境建议使用 Authorization 请求头方式。如果必须使用 Token 参数， 请确保启用 TLS 以防止 Token 以明文传输。

协议级认证

协议 认证方式
HTTP REST API Basic Auth 请求头或 Token 查询参数
Arrow Flight (gRPC) gRPC metadata 请求头 authorization: Basic ...
MySQL 线协议 原生 MySQL 认证握手
OpenTelemetry (OTLP) gRPC metadata 请求头或 HTTP Basic Auth

安全最佳实践

✅ 推荐做法
  • 在生产环境中始终启用认证。不启用认证会将所有数据和管理操作暴露给任何具有网络访问权限的人。
  • 为所有连接启用 TLS。没有 TLS，凭据和数据将以明文传输，可能在网络上被截获。
  • 在零信任环境中使用双向 TLS（mTLS）。要求客户端证书以验证连接的双方。
  • 遵循最小权限原则。为用户分配允许其完成工作的最低限制角色。仪表盘使用 reader，数据采集使用 writer，只有运维人员使用 admin。
  • 按用户限定数据库访问范围。仅授予每个用户所需的数据库访问权限，而不是使用通配符（*）访问。
  • 定期轮换凭据。按计划更新密码，并在团队成员离职时立即撤销访问权限。
  • 启用审计日志。维护所有操作的完整记录，用于安全监控、事件调查和合规。
  • 监控认证失败。对审计日志中重复的 AUTH_FAILURE 事件设置告警，以检测暴力破解尝试。
  • 限制网络访问。使用防火墙和网络策略限制哪些主机可以访问 MynaDB 端口（7000、7001、7002）。
  • 保护配置文件。config.toml 文件包含密码，应仅允许 MynaDB 服务用户读取（chmod 600）。
  • 使用环境变量存储密钥。不要将密码直接存储在配置文件中，使用环境变量引用：password = "${MYNADB_ADMIN_PASSWORD}"。
  • 将审计日志存储在独立存储上。将审计日志存储在不同的卷上或传输到远程系统，以防数据节点被入侵时日志被篡改。

示例：生产环境安全配置

toml 
# Production security configuration

[auth]
enabled = true
username = "admin"
password = "${MYNADB_ADMIN_PASSWORD}"

[tls]
enabled = true
cert_file = "/etc/mynadb/certs/server.crt"
key_file = "/etc/mynadb/certs/server.key"
ca_file = "/etc/mynadb/certs/ca.crt"
mutual_auth = true

[audit]
enabled = true
log_file = "/var/log/mynadb/audit.log"
rotation = "daily"
max_files = 90

[logging]
level = "info"
log_dir = "/var/log/mynadb"
