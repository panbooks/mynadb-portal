新功能

持续聚合与降采样

持续聚合为 MynaDB 带来了类似 TimescaleDB 的物化视图功能。只需定义一次查询，MynaDB 就会在新数据到达时自动保持结果更新。查询引擎会透明地重写传入的查询，尽可能从 预计算的聚合中读取数据，从而大幅降低仪表板和监控工作负载的查询延迟。

💡 为什么选择持续聚合？

无需在每次仪表板刷新时运行昂贵的 GROUP BY time_bucket(...) 查询， 只需定义一次聚合，让 MynaDB 以增量方式维护结果。对于大数据集上的时间分桶查询， 典型加速比为 10 倍到 100 倍。

创建聚合

使用类 SQL 的 CREATE CONTINUOUS AGGREGATE 语法在时间序列数据上定义物化视图。 每个聚合都需要一个 time_bucket() 调用和至少一个聚合函数。

sql 
-- 创建 CPU 指标的 5 分钟汇总
CREATE CONTINUOUS AGGREGATE cpu_5m
  WITH (refresh_interval = '1m')
AS
  SELECT
    time_bucket(time, 300000000000, 0) AS bucket,
    host,
    AVG(usage) AS avg_usage,
    MAX(usage) AS max_usage,
    MIN(usage) AS min_usage,
    COUNT(*) AS sample_count
  FROM cpu
  GROUP BY time_bucket(time, 300000000000, 0), host;

定义的关键要素：

要素 说明
cpu_5m 持续聚合的名称（在数据库内必须唯一）
refresh_interval 物化数据的刷新频率（例如 1m、5m、1h）
time_bucket() 必需的分桶函数，定义输出粒度
聚合函数 AVG、MAX、MIN、SUM、COUNT、first_value、last_value

使用 API

也可以通过 HTTP API 创建聚合：

bash 
# 通过 API 创建持续聚合
curl -X POST http://localhost:7000/aggregates/mydb \
  -H "Content-Type: application/json" \
  -d '{
    "name": "cpu_5m",
    "query": "SELECT time_bucket(time, 300000000000, 0) AS bucket, host, AVG(usage) AS avg_usage, MAX(usage) AS max_usage, MIN(usage) AS min_usage, COUNT(*) AS sample_count FROM cpu GROUP BY time_bucket(time, 300000000000, 0), host",
    "refresh_interval_secs": 60
  }'

自动降采样

持续聚合支持自动降采样策略，可随时间推移降低数据粒度。 这对于在保持长期可见性的同时管理存储成本至关重要。定义不同粒度的多个聚合， 即可构建分层汇总管道。

sql 
-- 第 1 层：原始数据（通过保留策略保留 7 天）
-- 1 分钟分辨率

-- 第 2 层：5 分钟汇总（保留 30 天）
CREATE CONTINUOUS AGGREGATE cpu_5m
  WITH (refresh_interval = '1m')
AS
  SELECT
    time_bucket(time, 300000000000, 0) AS bucket,
    host,
    AVG(usage) AS avg_usage,
    MAX(usage) AS max_usage,
    MIN(usage) AS min_usage,
    first_value(usage, time) AS first_usage,
    last_value(usage, time) AS last_usage,
    COUNT(*) AS sample_count
  FROM cpu
  GROUP BY time_bucket(time, 300000000000, 0), host;

-- 第 3 层：1 小时汇总（保留 1 年）
CREATE CONTINUOUS AGGREGATE cpu_1h
  WITH (refresh_interval = '5m')
AS
  SELECT
    time_bucket(time, 3600000000000, 0) AS bucket,
    host,
    AVG(usage) AS avg_usage,
    MAX(usage) AS max_usage,
    MIN(usage) AS min_usage,
    COUNT(*) AS sample_count
  FROM cpu
  GROUP BY time_bucket(time, 3600000000000, 0), host;

分层方案遵循常见模式：

层级 粒度 保留期 使用场景
原始数据 原始精度（如 1 秒） 7 天 调试、事件调查
第 1 层聚合 5 分钟 30 天 近期仪表板、告警
第 2 层聚合 1 小时 1 年 容量规划、趋势分析
提示： 将持续聚合与保留策略结合使用， 可以在保留降采样摘要的同时自动过期原始数据。

查询重写

当 query_rewriting_enabled 设为 true 时，查询引擎会自动 检测传入查询是否可由持续聚合满足，并将查询计划重写为从物化视图读取数据， 而非扫描原始数据。

工作原理

  1. 到达一个在源表上使用 time_bucket() 或 GROUP BY time() 的查询。
  2. 查询重写器检查该表是否存在具有匹配或更粗粒度桶大小的持续聚合。
  3. 如果聚合的桶大小能整除请求的桶大小，查询将被重写为从聚合中读取。
  4. 原始查询中的聚合函数将被转换为对预聚合列的操作。
sql 
-- 原始查询（用户编写）
SELECT
  time_bucket(time, 3600000000000, 0) AS hour,
  host,
  AVG(usage) AS avg_cpu
FROM cpu
WHERE time >= NOW() - INTERVAL '24 hours'
GROUP BY time_bucket(time, 3600000000000, 0), host;

-- 重写后的查询（引擎内部执行）
-- 从 5 分钟聚合中读取并重新聚合为 1 小时
SELECT
  time_bucket(bucket, 3600000000000, 0) AS hour,
  host,
  SUM(avg_usage * sample_count) / SUM(sample_count) AS avg_cpu
FROM cpu_5m
WHERE bucket >= NOW() - INTERVAL '24 hours'
GROUP BY time_bucket(bucket, 3600000000000, 0), host;

重写规则

原始函数 重写为 前提条件
AVG(col) SUM(avg_col * count) / SUM(count) 聚合必须包含 AVG 和 COUNT
SUM(col) SUM(sum_col) 聚合必须包含 SUM
COUNT(*) SUM(count_col) 聚合必须包含 COUNT
MIN(col) MIN(min_col) 聚合必须包含 MIN
MAX(col) MAX(max_col) 聚合必须包含 MAX

您可以使用 /explain 端点来验证查询重写是否生效：

bash 
# 检查查询是否使用了持续聚合
curl "http://localhost:7000/explain/mydb?sql=SELECT+time_bucket(time,3600000000000,0)+AS+hour,+host,+AVG(usage)+FROM+cpu+GROUP+BY+1,2&verbose=true"

# 在逻辑计划中查找 "ContinuousAggregate: cpu_5m"

配置

在 config.toml 中启用和调优持续聚合：

toml 
[continuous_aggregates]
# 启用持续聚合功能
enabled = true

# 后台任务检查需要刷新的聚合的频率（秒）
refresh_check_interval_secs = 60

# 最大并发刷新聚合数
max_concurrent_refreshes = 4

# 有利时自动将查询重写为使用聚合
query_rewriting_enabled = true
参数 默认值 说明
enabled false 持续聚合功能的主开关。禁用时，现有聚合不会刷新且查询重写不活跃。
refresh_check_interval_secs 60 后台刷新检查之间的间隔（秒）。较低的值会提高数据新鲜度，但会增加 CPU 使用率。
max_concurrent_refreshes 4 并行运行的最大聚合刷新任务数。对于拥有大量聚合和可用 CPU 核心的系统，可以适当增加。
query_rewriting_enabled true 启用后，查询引擎会自动将符合条件的查询重写为使用物化视图。可禁用以强制查询原始数据。
注意： 在集群模式下，只有选举出的 Leader 节点运行后台刷新任务。 查询重写在所有节点上都处于活跃状态。详见企业版集群。

API 端点

通过 REST API 管理持续聚合：

端点 方法 说明
/aggregates/:db GET 列出数据库的所有持续聚合
/aggregates/:db POST 创建新的持续聚合
/aggregates/:db/:name DELETE 删除持续聚合及其物化数据
/aggregates/:db/:name/refresh POST 手动触发立即刷新

列出聚合

bash 
# 列出所有持续聚合
curl http://localhost:7000/aggregates/mydb

# 响应
{
  "aggregates": [
    {
      "name": "cpu_5m",
      "source_table": "cpu",
      "bucket_interval_ns": 300000000000,
      "refresh_interval_secs": 60,
      "state": "active",
      "last_refresh": "2026-02-13T10:15:00Z",
      "watermark": 1739440500000000000
    },
    {
      "name": "cpu_1h",
      "source_table": "cpu",
      "bucket_interval_ns": 3600000000000,
      "refresh_interval_secs": 300,
      "state": "active",
      "last_refresh": "2026-02-13T10:10:00Z",
      "watermark": 1739438400000000000
    }
  ],
  "count": 2
}

创建聚合

bash 
# 创建内存指标的 1 小时汇总
curl -X POST http://localhost:7000/aggregates/mydb \
  -H "Content-Type: application/json" \
  -d '{
    "name": "memory_1h",
    "query": "SELECT time_bucket(time, 3600000000000, 0) AS bucket, host, AVG(used) AS avg_used, MAX(used) AS max_used, AVG(total) AS avg_total FROM memory GROUP BY time_bucket(time, 3600000000000, 0), host",
    "refresh_interval_secs": 300
  }'

# 响应
{
  "status": "created",
  "name": "memory_1h",
  "message": "Continuous aggregate created. First refresh will begin within 60 seconds."
}

删除聚合

bash 
# 删除持续聚合
curl -X DELETE http://localhost:7000/aggregates/mydb/cpu_5m

# 响应
{
  "status": "deleted",
  "name": "cpu_5m",
  "message": "Continuous aggregate and materialized data removed."
}

手动刷新

bash 
# 触发立即刷新
curl -X POST http://localhost:7000/aggregates/mydb/cpu_5m/refresh

# 响应
{
  "status": "refreshing",
  "name": "cpu_5m",
  "message": "Refresh started. New data will be available shortly."
}

监控

通过列表端点和服务器日志跟踪持续聚合的健康状况和性能。

聚合状态

状态 说明
active 聚合运行正常，正在按计划刷新。
refreshing 正在进行刷新。
error 上次刷新失败。请查看日志了解详情。系统将在下一个间隔重试。
disabled 聚合存在但未被刷新（例如，配置中禁用了该功能）。

关键指标

列表端点为每个聚合返回有用的监控字段：

  • last_refresh - 最近一次成功刷新的时间戳。过期的时间戳可能表示存在问题。
  • watermark - 已物化的最高时间戳（纳秒）。此时间点之后的数据尚未包含在聚合中。
  • state - 聚合的当前状态（参见上表）。

日志消息

刷新任务以 info 级别输出结构化日志。需要关注的关键消息： 

# 刷新成功
INFO  continuous_aggregates::job: Refreshed aggregate name="cpu_5m" rows=1842 duration_ms=234 watermark=1739440500000000000

# 刷新错误
ERROR continuous_aggregates::job: Failed to refresh aggregate name="cpu_5m" error="table not found: cpu" retries=3

# 查询重写已应用
DEBUG continuous_aggregates::query_rewriter: Rewrote query to use aggregate original_table="cpu" aggregate="cpu_5m"
