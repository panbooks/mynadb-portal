Admin Dashboard Guide

A comprehensive tour of the MynaDB Admin Console -- from cluster health monitoring to interactive SQL queries. Learn how the dashboard works, what each page does, and how to get started managing your cluster through the browser.

Overview

The MynaDB Admin Console is a standalone React web application that provides full visibility and control over your MynaDB cluster. It runs as a separate admin node that connects to the meta service for cluster discovery and forwards requests to the appropriate data nodes.

Architecture

The admin node sits alongside your cluster and communicates with two layers:

text 
  Browser (Admin UI)
        |
        v
  +------------------+
  |   Admin Node     |  Port 18080
  |   (Axum HTTP)    |
  +--------+---------+
           |
     +-----+-----+
     |           |
     v           v
  +------+   +--------+    +--------+
  | Meta |   | Data 1 |    | Data 2 |  ...
  | Svc  |   | Node   |    | Node   |
  +------+   +--------+    +--------+

  Meta Service:  Cluster discovery, node registry, leader election
  Data Nodes:    Query execution, storage, compaction
  Admin Node:    Web UI, request forwarding, event streaming
  • Meta Service -- The admin node queries meta endpoints to discover all registered data nodes, their roles, states, and table assignments
  • Data Nodes -- Queries, storage operations, compaction, and flush commands are forwarded directly to the appropriate data node's HTTP API
  • Event Streaming -- The admin node receives real-time cluster events (node joins, leader changes, table reassignments) and streams them to the browser via WebSocket

Dashboard

The Dashboard is the landing page of the Admin Console. It provides a high-level overview of your cluster's health and key statistics at a glance.

Cluster Health

The top section displays the overall cluster status:

  • Cluster Status -- Healthy (all nodes responding) or Degraded (one or more nodes unreachable)
  • Meta Leader -- The current meta leader node ID and its Raft term number

Node Count Cards

Four summary cards show the count of each node type in the cluster:

Card Shows
Meta Nodes Total meta service instances (active/inactive)
Router Nodes Total router instances for query routing
Data Nodes Total data nodes with primary/standby/retiring/drained breakdown
Tables Total registered tables and partitions across the cluster

Data Node Breakdown

Below the cards, a detailed breakdown shows data node states:

  • Primary -- Active nodes accepting writes and queries
  • Standby -- Healthy nodes ready to take over if a primary fails
  • Retiring -- Nodes in the process of draining tables during scale-in
  • Drained -- Nodes that have completed table migration and can be safely removed

Quick Actions

The dashboard includes quick action cards that link to the most common operational pages: Nodes management, Tables overview, and Scale-In wizard.

Navigation

The Admin Console uses a sidebar navigation layout. All pages are organized into two groups:

Main Pages

Page Path Purpose
Dashboard / Cluster health overview, node counts, quick actions
Query Console /query Monaco SQL editor with execute, explain, and analyze
Storage Browser /storage Tree view of catalogs, schemas, tables, and Parquet files
Nodes /nodes Node list with role/state filters and search
Node Detail /nodes/:id Individual node health, stats, and management actions
Tables /tables Table list with ownership, size, and reassignment
Scale-In /scale-in Multi-step wizard for cluster scale-in operations
Events /events Real-time cluster event stream with filters

Operations Pages

Page Path Purpose
Meta Operations /meta-operations Meta cluster status, elections, distribution config
Data Operations /data-operations Bulk flush, compact, and cache operations across nodes
Debug Tools /debug Test data generation, profiling, and cache management
Settings /settings Admin node configuration and cluster info display

Getting Started Walkthrough

Follow these six steps to familiarize yourself with the Admin Console after launching your cluster:

💡 Prerequisites

Make sure your cluster is running with an admin node configured. See Admin Console Configuration or use the dev cluster quick start.

Step 1: Check Cluster Health

Open http://localhost:18080 in your browser. The Dashboard loads automatically, showing the cluster status, meta leader, and node counts. Verify that the cluster is Healthy and all expected nodes appear.

Step 2: Generate Test Data

Navigate to Debug Tools in the sidebar. Use the test data generation card to write sample metrics (CPU, memory, disk, network) into a database. Configure the database name, date range, and number of data points, then click Generate.

Step 3: Run Your First Query

Go to the Query Console. Select your catalog from the dropdown, then try a simple query:

sql 
-- List all tables in the selected catalog
SHOW TABLES;

-- Query the test data you just generated
SELECT * FROM default.cpu ORDER BY time DESC LIMIT 10;

Press Ctrl+Enter (or Cmd+Enter on macOS) to execute. Results appear in a sortable, paginated table below the editor.

Step 4: Browse Storage

Open the Storage Browser and expand the catalog tree. Navigate through schemas and tables to see partition details, file counts, and storage sizes. Click on a Parquet file to inspect its row groups, compression, and column statistics.

Step 5: Monitor Events

Visit the Events page to see the real-time cluster event stream. Events like node joins, table registrations, and leader elections appear as they happen. Use the type and time filters to narrow the view.

Step 6: Review Settings

Check the Settings page to confirm your admin node configuration, including the HTTP port, meta endpoints, and cluster information.

Key Concepts

These definitions will help you understand the terminology used throughout the Admin Console:

🗃

Catalog

The top-level namespace, corresponding to the database name in the write URI (/write/{db}). Each catalog contains one or more schemas.

📁

Schema

A grouping within a catalog, derived from the service tag (ServiceAsSchema mode) or fixed as default (ServiceAsTable mode).

📅

Partition

Data within a table is partitioned by date (date=YYYY-MM-DD). Partitions enable efficient time-range pruning during queries.

📊

Compaction Level

Parquet files progress through compaction levels: L0 (recent writes), L1 (first merge), L2+ (higher merges). Higher levels have fewer, larger files for better query performance.

📝

WAL (Write-Ahead Log)

Incoming writes are recorded in the WAL for durability before being stored in the in-memory cache. The WAL is replayed on restart to recover unflushed data.

🌐

Node States

Data nodes transition between states: Active (serving traffic), Inactive (unreachable), Draining/Retiring (migrating tables), Drained (safe to remove).

🔗

Table Ownership

Each table is assigned to a primary data node that handles its writes and compaction. Tables can be reassigned to other nodes for load balancing or scale-in.

📈

Scale-In

The process of gracefully reducing cluster size by retiring data nodes. Tables are migrated to remaining nodes before the retiring node is drained.

Configuration

The admin node is configured via a TOML configuration file. Here is a typical setup:

toml 
[admin]
# Enable the admin console
enabled = true

# HTTP host and port for the admin web interface
host = "0.0.0.0"
port = 18080

# Meta service endpoints for cluster discovery
# List multiple endpoints for high availability
meta_endpoints = [
    "http://127.0.0.1:6000",
    "http://127.0.0.1:6100",
    "http://127.0.0.1:6200"
]

Configuration Options

Option Type Default Description
enabled bool false Enable or disable the admin console
host string "0.0.0.0" Bind address for the admin HTTP server
port u16 18080 HTTP port for the admin web interface
meta_endpoints array [] List of meta service URLs for cluster discovery

Dev Cluster Quick Start

The fastest way to try the admin console is with the local development cluster:

bash 
# Start the full local cluster (meta x3, data x2, router, admin)
goreman -f dev/Procfile.cluster-local start

# Default ports:
#   Meta nodes:  6000, 6100, 6200
#   Data nodes:  5000, 5100
#   Router:      7000
#   Admin:       18080

# Open the admin console
open http://localhost:18080
