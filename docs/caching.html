Multi-Level Caching

Perfect recall, every time. MynaDB implements a sophisticated multi-level caching architecture — like the myna bird's remarkable memory — to maximize query performance. This guide explains the different cache layers and how to configure them.

New in v0.2.8: Memory-based Metadata Manager with automatic persistence, Parquet Disk Cache for Object Store, and Memtable for immediate query visibility of fresh writes.

Cache Layers Summary

Cache Layer Storage Purpose Performance Impact
Metadata Manager Memory + JSON Catalog, schema, table, partition, file metadata Eliminates directory scans (10-100x faster planning)
Query Result Cache Memory SQL query results Instant response for repeated queries
Memtable Memory Fresh writes not yet flushed Immediate query visibility (no flush required)
Parquet Disk Cache Local SSD Object Store files 10-100x faster than network fetch

1. Metadata Manager

The Metadata Manager maintains a complete catalog of all databases, schemas, tables, partitions, and files in memory. This eliminates expensive directory scans and object store listings during query planning.

Architecture

text 
┌─────────────────────────────────────────────────────────────┐
│                    Metadata Manager                          │
│                                                              │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────────────┐  │
│  │  Catalogs   │  │   Schemas   │  │      Tables         │  │
│  │  HashMap    │  │   HashMap   │  │      HashMap        │  │
│  └─────────────┘  └─────────────┘  └─────────────────────┘  │
│                                                              │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────────────┐  │
│  │ Partitions  │  │    Files    │  │   Partition→Files   │  │
│  │  HashMap    │  │   HashMap   │  │   Reverse Index     │  │
│  └─────────────┘  └─────────────┘  └─────────────────────┘  │
│                                                              │
│         ↓                ↓                   ↓               │
│  ┌─────────────────────────────────────────────────────────┐ │
│  │             JSON Snapshot (Periodic Save)               │ │
│  │              catalog_snapshot.json                       │ │
│  └─────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────┘

What's Stored

Entity Key Tracked Information
CatalogInfo catalog_name Name, created_at, schema_count, table_count, total_bytes
SchemaInfo (catalog, schema) Name, catalog, table_count, total_bytes
TableInfo (catalog, schema, table) Name, total_rows, total_bytes, partition_count, file_count, min/max timestamp
PartitionInfo (catalog, schema, table, date) Partition key/value, total_rows, total_bytes, file_count, min/max timestamp
FileInfo file_path Path, level, size_bytes, row_count, row_groups, partition_key, etag

Key Features

  • Zero Directory Scans: All metadata in memory, no filesystem calls during query planning
  • Automatic Persistence: Periodic save to catalog_snapshot.json for crash recovery
  • Lazy Registration: New catalogs/schemas/tables auto-created when files are registered
  • Aggregated Statistics: Automatic rollup of row counts, bytes, timestamps up the hierarchy
  • Fast Partition Pruning: Query list_partitions_in_range() for date-filtered queries

2. Query Result Cache

Caches the results of SQL queries to avoid re-executing expensive queries.

Configuration

toml 
[query_cache]
# Enable query result caching
enabled = true

# Maximum cache size in MB
max_size_mb = 256

# Maximum number of cached queries
max_entries = 1000

# Cache TTL in seconds (5 minutes)
ttl_secs = 300

# Only cache queries that take longer than this (ms)
min_query_time_ms = 10

How It Works

  • Cache key is computed from: database name + SQL query text
  • Results are cached only if query execution time exceeds min_query_time_ms
  • Cache is automatically invalidated when data is written to referenced tables
  • Entries expire after ttl_secs seconds

Cache Invalidation

bash 
# Clear only query result cache
curl -X POST http://localhost:7000/cache/clear/results

# Cache is automatically invalidated on:
# - Data writes to a table
# - Compaction of a table
# - Manual flush

3. Memtable (Immediate Query Visibility)

The Memtable provides immediate query visibility for fresh writes that haven't been flushed to Parquet yet. Previously, you had to call /flush before querying recent data. Now queries automatically include in-memory data.

How It Works

text 
┌─────────────────────────────────────────────────────────────┐
│                      Query Execution                         │
│                                                              │
│  SELECT * FROM cpu WHERE host = 'server01'                   │
│                                                              │
│         ┌──────────────┐          ┌──────────────┐           │
│         │MemtableExec  │          │ParquetExec   │           │
│         │(Fresh Data)  │          │(Parquet Files)│          │
│         └──────┬───────┘          └──────┬───────┘           │
│                │                         │                    │
│                └──────────┬──────────────┘                    │
│                           │                                   │
│                           ▼                                   │
│                   ┌───────────────┐                           │
│                   │  UnionExec    │                           │
│                   │ (Merge Both)  │                           │
│                   └───────────────┘                           │
│                           │                                   │
│                           ▼                                   │
│                      Query Results                            │
│              (Includes ALL data, flushed + unflushed)         │
└─────────────────────────────────────────────────────────────┘

Key Features

  • Zero-Copy Access: Reads directly from DataCache without copying (Arc-based batches)
  • Schema Alignment: Automatically aligns memtable schema with Parquet file schema
  • Filter Pushdown: Supports inexact filter pushdown for efficiency
  • Streaming Results: Uses RecordBatchStream to avoid collecting all data in memory

Query Behavior

bash 
# Write data (goes to in-memory cache)
curl -X POST http://localhost:7000/write/mydb \
  -d 'cpu,host=server01 usage=75.5'

# Query IMMEDIATELY - no flush needed!
# This returns the fresh data from memtable
curl 'http://localhost:7000/query/mydb' \
  -d 'SELECT * FROM cpu WHERE host = '\''server01'\'''

# Data is automatically included from:
# 1. In-memory cache (memtable) - fresh writes
# 2. Parquet files - flushed data

4. Parquet Disk Cache (Object Store)

When using Object Store backends (S3, MinIO, Aliyun OSS), MynaDB caches Parquet files on local SSD to dramatically reduce network latency. This is essential for production deployments using cloud storage.

Architecture

text 
┌─────────────────────────────────────────────────────────────┐
│                      Disk File Cache                         │
│                                                              │
│  ┌─────────────────────────────────────────────────────────┐ │
│  │                  In-Memory State                        │ │
│  │  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐  │ │
│  │  │   entries    │  │  total_size  │  │    stats     │  │ │
│  │  │   HashMap    │  │    (u64)     │  │ hits/misses  │  │ │
│  │  └──────────────┘  └──────────────┘  └──────────────┘  │ │
│  └─────────────────────────────────────────────────────────┘ │
│                              │                               │
│                              ▼                               │
│  ┌─────────────────────────────────────────────────────────┐ │
│  │                  Local SSD Storage                      │ │
│  │  /var/cache/mynadb/parquet/                             │ │
│  │    ├── abc123def456.parquet  (cached file)              │ │
│  │    ├── 789ghi012jkl.parquet  (cached file)              │ │
│  │    └── cache_index.json      (metadata)                 │ │
│  └─────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────┘

Configuration

toml 
[file_cache]
# Enable file caching (essential for Object Store backends)
enabled = true

# Cache directory on local SSD (use NVMe for best performance)
cache_dir = "/var/cache/mynadb/parquet"

# Maximum cache size (100 GB)
max_size_bytes = 107374182400

# Index save interval in seconds (5 minutes)
auto_save_interval_secs = 300

Cache Entry Metadata

Each cached file tracks comprehensive metadata for smart eviction:

json 
{
  "source_path": "s3://bucket/mydb/default/cpu/date=2024-01-15/L0_xxx.parquet",
  "cache_path": "/var/cache/mynadb/parquet/abc123.parquet",
  "size_bytes": 10485760,
  "etag": "\"d41d8cd98f00b204e9800998ecf8427e\"",
  "cached_at": 1705123456789,
  "last_accessed": 1705234567890,
  "access_count": 42,
  "partition_date": "2024-01-15"
}

Smart Eviction Strategy

The cache uses a priority-based eviction algorithm that considers multiple factors:

text 
Priority Score = Recency × Frequency × Partition_Bonus

Recency Factors:
  - Accessed in last hour:    100x
  - Accessed today:           50x
  - Accessed this week:       10x
  - Older:                    1x

Partition Date Bonus (time series optimization):
  - Today's partition:        10x
  - Yesterday's partition:    8x
  - This week's partition:    5x
  - This month's partition:   2x
  - Older partitions:         1x

Files with LOWER scores are evicted first.

Cache Operations

bash 
# Prefetch files for expected queries (warm cache before dashboard load)
curl -X POST http://localhost:7000/cache/prefetch \
  -H "Content-Type: application/json" \
  -d '{"paths": ["s3://bucket/mydb/default/cpu/date=2024-01-15/*"]}'

# Invalidate cache for specific path prefix (after compaction)
curl -X POST http://localhost:7000/cache/invalidate \
  -H "Content-Type: application/json" \
  -d '{"prefix": "s3://bucket/mydb/default/cpu/date=2024-01-15/"}'

# Clear entire disk cache
curl -X POST http://localhost:7000/cache/clear

# Verify cache integrity (check for orphaned entries)
curl http://localhost:7000/cache/verify

Best Practices

For High Query Performance

Metadata Manager

  • Let the system auto-populate metadata from writes
  • Pre-warm on startup for critical databases
  • Monitor partition counts for table growth

Query Result Cache

  • Enable for dashboard/reporting queries
  • Set TTL based on data freshness needs
  • Use min_query_time_ms to skip fast queries

Memtable

  • Auto-enabled, no configuration needed
  • Query immediately after writes
  • Adjust flush_interval_secs for memory vs freshness

Disk Cache

  • Use NVMe SSD for cache directory
  • Size to hold 7+ days of hot data
  • Monitor hit rate (target >90%)
