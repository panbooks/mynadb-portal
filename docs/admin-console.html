Admin Console

Web-based administration interface for managing and monitoring your MynaDB cluster. The Admin Console provides a unified dashboard for querying data, browsing storage, managing cluster nodes, and tracking query history -- all from your browser.

Overview

The MynaDB Admin Console is a standalone web application that connects to your cluster via the meta service. It provides full visibility into your databases, tables, partitions, and cluster health, along with a powerful SQL query editor for interactive data exploration.

📝

SQL Query Console

Built-in Monaco editor with SQL syntax highlighting, auto-complete, and tabular result display. Execute, explain, and analyze queries directly in the browser.

🗃

Storage Browser

Navigate your data hierarchy with a tree view of catalogs, schemas, tables, and partitions. Inspect Parquet file details and manage partitions.

🌐

Cluster Management

Monitor node status and health in real time. View the cluster event log with filtering, and track node joins, departures, and leader elections.

🕐

Query History

Save frequently used queries for quick access. Browse execution history with timing details, and review explain plans for past queries.

Configuration

Enable and configure the Admin Console in your config.toml. The admin node runs as a separate process that connects to one or more meta service endpoints for cluster discovery and forwards requests to the appropriate data nodes.

toml 
[admin]
# Enable the admin console
enabled = true

# Port for the admin web interface
port = 18080

# Meta service endpoints for cluster discovery
meta_endpoints = [
    "http://127.0.0.1:6000",
    "http://127.0.0.1:6100",
    "http://127.0.0.1:6200"
]

Configuration Options

Option Type Default Description
enabled bool false Enable or disable the admin console
port u16 18080 HTTP port for the admin web interface
meta_endpoints array [] List of meta service URLs for cluster discovery
💡 Meta Service Discovery

The admin console uses the meta service to discover all data nodes in the cluster. Provide at least one meta endpoint. For high availability, list multiple meta endpoints so the admin can fail over if one becomes unavailable.

SQL Query Console

The SQL Query Console is the primary interface for interacting with your data. It provides a full-featured code editor powered by Monaco (the same editor used in VS Code) with dedicated controls for executing, explaining, and analyzing queries.

Features

  • Monaco Editor -- Full SQL syntax highlighting, bracket matching, and auto-indentation
  • Execute Query -- Run SQL queries and view results in a sortable, paginated table
  • Explain Plan -- View the logical and physical execution plans for a query
  • Analyze Query -- See partition pruning details, file counts, and optimization summaries
  • Export Results -- Download query results as CSV or JSON
  • Saved Queries -- Save and organize frequently used queries for quick access
  • Query History -- Automatically records every executed query with timing and row counts

Database and Schema Selection

Use the catalog and schema dropdowns at the top of the console to set the query context. The console auto-discovers available databases and schemas from the cluster:

sql 
-- With catalog "mydb" selected, you can use short names:
SELECT * FROM default.cpu WHERE time > now() - INTERVAL '1 hour';

-- Or fully qualified names work from any context:
SELECT * FROM mydb.default.cpu WHERE time > now() - INTERVAL '1 hour';

Keyboard Shortcuts

Shortcut Action
Ctrl+Enter / Cmd+Enter Execute the current query
Ctrl+E / Cmd+E Explain the current query
Ctrl+S / Cmd+S Save the current query
Ctrl+Shift+A / Cmd+Shift+A Analyze the current query

Storage Browser

The Storage Browser provides a hierarchical view of all data stored in your MynaDB cluster. Navigate through catalogs, schemas, tables, and partitions to inspect storage details and manage your data.

Features

  • Tree Navigation -- Expand catalogs, schemas, and tables in a collapsible tree view
  • Table Details -- View table schema, column types, row counts, and storage size
  • Partition List -- Browse date partitions with file counts and size per partition
  • Parquet File Details -- Inspect individual Parquet files including row groups, compression, and column statistics
  • Partition Management -- Delete individual partitions or date ranges directly from the UI
  • Trigger Compaction -- Initiate compaction for a specific table or the entire catalog

Storage Hierarchy

text 
Cluster
 +-- mydb                          (catalog)
 |    +-- default                  (schema)
 |    |    +-- cpu                 (table)
 |    |    |    +-- date=2024-01-07  (partition, 3 files, 12.4 MB)
 |    |    |    +-- date=2024-01-08  (partition, 2 files, 8.1 MB)
 |    |    |    +-- date=2024-01-09  (partition, 1 file, 3.2 MB)
 |    |    +-- memory              (table)
 |    +-- service_1                (schema)
 |         +-- disk                (table)
 +-- otel                          (catalog)
      +-- default                  (schema)
           +-- otel_metrics        (table)
           +-- otel_logs           (table)

Compaction Controls

From the Storage Browser, you can trigger compaction at different levels:

bash 
# The admin console triggers these API calls on the data node:

# Compact an entire catalog
POST /compact/mydb

# Compact a specific measurement
POST /compact/mydb/cpu

Cluster Management

The Cluster Management view gives you real-time visibility into every node in your MynaDB cluster. Monitor node health, track events, and quickly identify issues.

Node List

The node list displays all registered data nodes with their current status:

Column Description
Node ID Unique identifier for the node
Address HTTP and Flight endpoint addresses
Status Online, Offline, or Degraded
Role Leader or Follower (for background task assignment)
Uptime Time since the node last registered with the meta service
Tables Number of tables assigned to this node

Event Log

The event log captures cluster-level events in real time. Use the filter controls to narrow events by type or time range:

  • Node Joined -- A data node registered with the meta service
  • Node Left -- A data node deregistered or timed out
  • Leader Elected -- A new leader was elected for background tasks
  • Table Created -- A new table was registered in the cluster
  • Compaction Started / Completed -- A compaction job began or finished
  • Retention Executed -- Expired partitions were deleted

Health Checks

The admin console periodically polls each data node's /health endpoint and displays aggregated cluster health. A node is marked as degraded if it fails to respond within the configured timeout, and offline if it misses multiple consecutive health checks.

⚠ Node Connectivity

The admin console must be able to reach each data node's HTTP port directly. Ensure there are no firewall rules or network policies blocking traffic between the admin node and data nodes.

Accessing the Console

Once the admin node is running, open the console in your browser:

text 
http://localhost:18080

The default port is 18080. If you changed the port in your configuration, use the configured value instead.

Authentication

If authentication is enabled on the data nodes, the admin console will prompt you for credentials on first access. The same username and password configured in the [auth] section of your data node configuration apply:

toml 
# Data node auth configuration
[auth]
enabled = true
username = "admin"
password = "secret"
🔒 Security Recommendation

In production environments, always enable authentication and access the admin console over HTTPS by placing it behind a reverse proxy such as Nginx or Caddy with TLS termination. See the Authentication & RBAC page for more details.

Dev Cluster Quick Start

To run the admin console as part of a local development cluster:

bash 
# Start the full local cluster (meta, data, router, admin)
goreman -f dev/Procfile.cluster-local start

# Default ports:
#   Meta nodes:  6000, 6100, 6200
#   Data nodes:  5000, 5100
#   Router:      7000
#   Admin:       18080

# Open the admin console
open http://localhost:18080

Events Monitoring

The Events page provides a real-time stream of cluster events delivered via WebSocket. Monitor node joins, departures, leader elections, table reassignments, and more as they happen. Filter events by type, time range, and source, and export them as JSON or CSV for analysis.

For a complete list of all 13 event types and detailed usage instructions, see the Events Monitoring section in the Cluster Operations Guide.

Scale-In Operations

The Scale-In page provides a guided three-step wizard for gracefully reducing the number of data nodes in your cluster. Choose from strategies like Least Loaded, First In, Last In, or HA Unit, preview the migration plan, and execute with real-time progress tracking.

For the full walkthrough of the scale-in wizard, see the Scale-In Operations section in the Cluster Operations Guide.

Debug Tools

The Debug Tools page provides utilities for test data generation, CPU and heap profiling, and cache management. Generate sample time-series metrics, download profiling snapshots, or prewarm and clear caches across your data nodes.

For detailed instructions on each debug tool, see the Debug Tools section in the Cluster Operations Guide.
