Cluster Operations Guide

Detailed instructions for managing your MynaDB cluster through the Admin Console. This guide covers node management, table operations, scale-in workflows, event monitoring, meta and data operations, debug tools, and settings.

Nodes Management

The Nodes page (/nodes) displays all registered cluster nodes in a card-based grid layout. Each card shows the node ID, advertise address, and color-coded badges for role, state, and mode.

Filters

Use the filter bar at the top to narrow the node list:

Filter Options Description
Search Free text Filter nodes by ID or address
Role Meta, Router, Data, Compactor Filter by the node's cluster role
State Active, Inactive, Draining, Retiring, Drained Filter by operational state
Mode Primary, Standby Filter by node mode (data nodes only)

Node Cards

Each node card displays:

  • Node ID -- Unique identifier assigned at registration
  • Advertise Address -- The host:port other nodes use to reach this node
  • Role Badge -- Meta (blue), Router (green), Data (purple), Compactor (orange)
  • State Badge -- Active (green), Inactive (gray), Draining (yellow), Retiring (orange), Drained (red)
  • Mode Badge -- Primary or Standby (data nodes only)
  • Zone -- Availability zone or rack assignment
  • Tables Owned -- Number of tables assigned to this node
  • Last Heartbeat -- Relative timestamp of the last health check

Click any node card to navigate to the Node Detail page.

Node Detail Page

The Node Detail page (/nodes/:id) provides full information about a single node and management actions:

Action Description
Flush Node Force the node to flush its in-memory write cache to Parquet files
Compact Node Trigger compaction of all tables on this node
Clear Node Cache Clear the node's metadata and query result caches
Retire Node Begin retiring the node -- migrates tables to other nodes, then drains
Cancel Retire Cancel an in-progress retirement (available during draining/retiring states)

When a node is in a retiring or draining state, the detail page shows a progress tracker with the number of tables migrated and remaining.

⚠ Retire vs Remove

Always use the Retire action instead of abruptly stopping a data node. Retiring gracefully migrates table ownership to other healthy nodes before the node shuts down, preventing data unavailability.

Tables Management

The Tables page (/tables) lists all tables registered in the cluster with key metadata and management actions.

Table List

The table list supports search, filtering by catalog and schema, and pagination (20 tables per page). Each row displays:

Column Description
Name Fully qualified name (catalog.schema.table)
Size Total storage size across all partitions
Rows Approximate total row count
Partitions Number of date partitions
Owner Node The primary data node responsible for this table (clickable link to node detail)
Replica Count Number of replica copies across nodes
Actions Reassign button to move the table to a different node

Reassign Table

Click the Reassign button on any table row to open a dialog where you can select a target data node. Table reassignment transfers ownership so that the new node handles writes and compaction for that table.

Common reasons to reassign a table:

  • Load balancing -- Move tables from overloaded nodes to less busy ones
  • Maintenance -- Migrate tables off a node before performing maintenance
  • Scaling -- Redistribute tables after adding new data nodes
  • Recovery -- Re-home tables from a failed node to healthy nodes

Scale-In Operations

The Scale-In page (/scale-in) provides a guided three-step wizard for gracefully reducing the number of data nodes in your cluster.

Step 1: Configure

In the configuration step, you specify:

  • Node Count -- How many nodes to remove from the cluster
  • Strategy -- The algorithm for selecting which nodes to retire:
Strategy Description
Least Loaded Retire the nodes with the fewest tables (minimizes migration work)
First In Retire the oldest nodes by registration time
Last In Retire the most recently added nodes
HA Unit Retire nodes from the same availability zone to maintain zone balance

Optionally, you can manually select specific node IDs instead of using an automatic strategy. The wizard shows recommendations based on the current cluster state.

Step 2: Preview

After configuring, the wizard performs a dry-run and displays the scale-in plan:

  • Plan Summary -- Total nodes to retire, tables to migrate, estimated duration
  • Nodes to Retire -- List of selected nodes with their current table counts
  • Execution Phases -- Ordered list of migration steps showing which tables move where

Review the plan carefully before proceeding. You can go back to Step 1 to adjust the configuration.

Step 3: Execute

Click Execute to begin the scale-in operation. The page shows real-time progress:

  • Progress Bar -- Overall completion percentage
  • Phase Status -- Current phase and its status (pending, in-progress, completed)
  • Per-Node Progress -- Tables migrated vs total for each retiring node
  • Operation Status -- Running, Completed, or Failed with error details

You can Cancel a running scale-in operation. Cancellation stops future migrations but does not roll back already-completed table moves.

💡 Scale-In Best Practices

Scale in one or two nodes at a time rather than many at once. This limits the amount of in-flight table migration and reduces impact on query performance. See Cluster Scaling for architectural details.

Events Monitoring

The Events page (/events) displays a real-time stream of cluster events delivered via WebSocket.

Real-Time Stream

Events appear as cards in reverse chronological order. Each event card shows:

  • Timestamp -- Relative time (e.g., "2 minutes ago") with full ISO timestamp on hover
  • Type Icon -- Color-coded icon indicating the event category
  • Source -- The node or component that generated the event
  • Severity -- Visual indicator for info, warning, or error events
  • Details -- Event-specific data payload

Connection Status

A status indicator shows whether the WebSocket connection to the admin node is active. If the connection drops, the page automatically attempts to reconnect. Use Pause / Resume to temporarily stop the live stream without disconnecting.

Filters

Filter Options
Time Range All Time, Last 1 Hour, Last 24 Hours, Last 7 Days
Event Type Select from the event types listed below
Event Source Filter by originating node or component

Event Types

Event Type Description
NodeJoined A node registered with the meta service
NodeLeft A node deregistered or timed out
NodeStateChanged A node transitioned between states (Active, Retiring, Drained, etc.)
LeaderChanged A new meta leader was elected
TableReassigned A table's ownership was moved to a different node
TableOwnerSet A table was assigned an owner for the first time
TableOwnerRemoved A table's owner was removed (node failure or deregistration)
CacheInvalidate A cache invalidation message was broadcast to data nodes
SchemaChanged A table's schema was modified (new columns, type changes)
FlushForRecovery A node was instructed to flush its cache for WAL recovery
DistributionConfigChanged The table distribution configuration was updated
TablesCacheRefresh The tables cache was refreshed on one or more nodes
Unknown An unrecognized event type (forward compatibility)

Export

Use the export buttons to download the current event list:

  • JSON -- Full event objects with all metadata
  • CSV -- Tabular format for spreadsheet analysis

Enable Browser Notifications (bell icon) to receive desktop alerts for important events like node failures or leader changes.

Meta Operations

The Meta Operations page (/meta-operations) provides visibility and control over the meta service cluster.

Meta Cluster Status

The status section displays:

  • Cluster Health -- Healthy if a majority of meta peers are reachable
  • Current Term -- The Raft consensus term number
  • Commit Index -- Latest committed log entry index
  • Meta Leader -- Node ID and address of the current leader

Meta Peers

A table lists all meta service instances with their addresses, roles (Leader/Follower), and connectivity status.

Leader Elections

Two buttons allow you to manually trigger leader election actions:

  • Campaign -- Force the current meta leader to start a new election
  • Resign -- Ask the current leader to step down, triggering a new election
⚠ Election Caution

Manually triggering elections causes a brief disruption while a new leader is elected. Only use these during maintenance windows or when troubleshooting leadership issues.

Distribution Configuration

The distribution configuration controls how new tables are assigned to data nodes. You can view and modify:

  • Scope -- Apply distribution rules at the catalog, schema, or node level
  • Replication Factor -- Number of replica copies for table data
  • Placement Strategy -- Algorithm for choosing which nodes receive new tables

Data Operations

The Data Operations page (/data-operations) provides bulk and per-node operations for all data nodes in the cluster.

Summary Cards

At the top, summary cards show the total number of data nodes, active nodes, and the last operation status.

Bulk Operations

Action Description
Flush All Trigger a cache flush on all active data nodes simultaneously
Compact All Trigger compaction on all active data nodes simultaneously

Per-Node Operations

Below the bulk actions, a table lists each data node with individual action buttons:

Action Description
Flush Flush the node's in-memory write cache to Parquet files on disk/object store
Compact Run compaction on all tables owned by this node, merging small L0 files into larger L1/L2 files
Clear Cache Clear the node's metadata cache and query result cache to free memory

Each action button shows a loading spinner while the operation is in progress. State badges on each row indicate the node's current operational status.

Debug Tools

The Debug Tools page (/debug) provides utilities for testing, profiling, and cache management.

Test Data Generation

Generate sample time-series data for testing and demonstration. Configure:

Option Description
Data Types CPU, Memory, Disk, Network metrics (select one or more)
Database Target catalog name for the generated data
Date Range Start and end dates for the generated time series
Points Number of data points to generate per metric type

Click Generate to write the test data via the data node's write API. The data is immediately queryable in the Query Console.

Profiling

Download profiling snapshots from the data nodes:

  • Heap Profile -- Memory allocation snapshot showing what is consuming memory
  • CPU Profile -- CPU usage profile for identifying hot code paths

Profiles are downloaded as files that can be analyzed with standard profiling tools like pprof or flamegraph generators.

Cache Management

  • Prewarm Cache -- Load metadata for a specific catalog into the data node's cache proactively, reducing first-query latency
  • Clear All Caches -- Drop all cached metadata and query results across data nodes to free memory

Settings

The Settings page (/settings) displays the current admin node configuration in a read-only view.

Information Displayed

  • Admin Configuration -- HTTP host, port, meta endpoints
  • Cluster Info -- Cluster ID, deployment mode
  • Network -- Bind addresses and connectivity details
  • Node Counts -- Summary of registered meta, router, and data nodes

Use the Refresh button to reload the configuration from the admin node.

💡 Configuration Changes

The Settings page is read-only. To change the admin node configuration, edit the TOML config file and restart the admin node. See Admin Guide - Configuration for available options.
